# not4k — PRD 작성 전 미결정 사항 검토

전체 설계 문서(설계 16개 + 리서치 6개)를 교차 검토하여, PRD 작성 전에 결정이 필요한 항목을 분류하였다.

---

## A. PRD 작성 전 반드시 결정해야 하는 사항

PRD의 스코프와 요구사항을 정의하는 데 직접적으로 영향을 주는 미결정 항목.

### ~~1. 기술 스택~~ → **해결됨**

React 19 + PixiJS v8 + Supabase + Web Audio API (AudioContext)로 확정. 상세 검토는 `tech-stack.md`를 참고.

### ~~2. 차트 데이터 포맷~~ → **해결됨**

JSON 포맷으로 확정. Supabase Storage에 정적 파일로 저장하고 `fetch()`로 가져온다. 상세는 `tech-stack.md`를 참고.

### ~~3. MVP 스코프 확정~~ → **해결됨**

Phase A(프로토타입)와 Phase B(알파)의 기능적 스코프를 확정. 게임 + 차트 에디터 + 공유 패키지의 Phase별 포함/제외 기능을 정의. 상세는 `mvp-scope.md`를 참고.

### ~~4. 오디오 동기화 전략~~ → **해결됨**

`AudioContext.currentTime`을 마스터 클럭으로 사용. 음원은 전체 로드(`AudioBuffer`) 방식. 상세는 `tech-stack.md`를 참고.

### ~~5. 타깃 유저 정의~~ → **해결됨**

#### 1차 타깃: 기존 노트낙하형 리듬게임 유저 중 신체적 부하로 좌절한 유저

osu!mania, DJMAX 등 4키+ 리듬게임 경험자 중 고난이도에서 **손가락/손목의 물리적 한계**에 막힌 유저. not4k가 이들에게 제공하는 것:

| 기존 4키의 문제 | not4k의 해법 |
|---------------|-------------|
| 물리적 속도·지구력이 난이도 천장 | 신체 부하 대신 **인식 부하**(손배치 전환 판단)로 난이도 상승 |
| 특정 손가락에 부하 집중 (약지·소지) | 8키 분산 → 손가락 사용이 **균등하게 분산** |
| 배치가 불편해도 강제 | 원하는 손가락을 선택 가능 → **배치 스트레스 없음** |

#### 2차 타깃: 리듬게임 입문자

리듬게임을 처음 접하는 유저도 not4k로 시작할 수 있도록 **진입 장벽을 낮추는 장치**를 마련한다:

- 튜토리얼 (Phase B): 연주 방법 → 복합 패턴 단계적 학습
- Lv.1~3: 전 노트 타입이 등장하지만, 패턴 밀도와 조합이 단순
- 옵저버 모드 (Phase B): 어려운 구간을 미리 연습 가능

단, **메인 타깃은 기존 리듬게임에 질린/좌절한 유저**이다. 입문자는 "이 게임으로도 시작할 수 있다"는 접근성을 보장하되, 게임 설계의 핵심 의사결정은 1차 타깃 기준으로 내린다.

#### 지역 타깃

한국 → 일본 → 글로벌 순. 다국어(ko/ja/en) 확정.

## B. PRD에 "미정" 또는 "추후 결정"으로 표기할 수 있는 사항

PRD 작성 자체를 막지는 않지만, 해당 섹션에 미결정임을 명시해야 하는 항목.

### 6. 난이도명 및 판정명

`glossary.md`에서 잠정적으로 결정:
- 난이도명: EASY / NORMAL / HARD (3단계)
- 판정명: Perfect / Great / Good / Bad / Miss

게임 컨셉이 구체화되면 변경 가능하다고 명시되어 있으므로, PRD에는 잠정 값을 기재하고 변경 가능성을 표기하면 된다.

### 7. 실력 지표 시스템 (볼포스류)

`scoring.md`에서 "플레이 데이터 축적 후 도입"으로 유보되었다. PRD에는 "향후 도입 예정, MVP 범위 외"로 표기하면 된다.

### 8. 시각 디자인 미정 항목

여러 문서에서 시각적 표현이 "미정"으로 남아 있다. PRD에는 요구사항만 기술하고, 구체적 디자인은 디자인 단계로 넘기면 된다.

- 트릴 구간의 시각 표현 방식 (`note-system.md`)
- 트릴 노트의 시각 표현 방식 (`note-system.md`)
- 더블 노트의 시각 표현 방식 (`note-system.md`)
- 릴리즈탭의 시각 표현 (`note-system.md`)
- 홀드 중 탭의 시각 표현 (`note-system.md`)
- 홀드 트릴의 시각 표현 (`note-system.md`)

### 9. 키 바인딩 관련 열린 질문

`keybinding.md`의 열린 질문 3건:

- **왼손잡이 키보드 배치 반전**: `game-core.md`의 미러 옵션(레인 반전)과 별개로, 키보드 배치 자체를 좌우 반전할 것인가?
  - 미러 옵션은 이미 확정. 키보드 배치 반전은 추가 기능이므로 "향후 검토"로 표기 가능.
- **전용 컨트롤러 지원**: 게임패드 미지원은 `game-core.md`에서 확정. 전용 컨트롤러(예: 아케이드 컨트롤러)는?
  - 초기 MVP에서는 키보드 전용으로 충분. "향후 검토"로 표기 가능.
- **비 약중검엄 배치 허용**: "제한하지 않되 약중검엄을 권장 프리셋으로 제공" 방향이 유력하나 확정 아님.
  - 유력한 방향을 PRD에 기재하되, 확정 전임을 표기.

### 10. 튜토리얼 구조의 결합 방식

`tutorial.md`에서 두 트랙의 결합이 TBD:
- 키 바인딩 튜토리얼 (물리적 키 배치 발견: `keybinding.md`)
- 노트 메커니즘 튜토리얼 (연주법 + 판정 학습: `tutorial.md`)

병행 또는 순차 진행 가능하며, 구현 단계에서 결정해도 된다.

### 11. 채보 제작 가이드라인

`difficulty-design.md`에서 미정:
- 등급 간 경계의 노트 밀도, BPM 범위, 트릴 비율 등의 정량적 기준.

이것은 게임 시스템 구현 후 플레이테스트를 통해 조정하는 것이므로, PRD에서는 "구현 후 수립"으로 표기하면 된다.

### 12. 피스 카탈로그 미완료

`review-document-clarity.md` 17번:
- `chart-design.md`의 피스 설계 개념 → `piece-definition.md`의 PP-000 개별 피스 변환이 미완료.
- 기존 PP 카탈로그(PP-001, PP-003~PP-011)는 정의되었으나, `chart-design.md`의 5개 피스 개념과 완전히 대응하지 않는 항목 존재.

PRD 수준에서는 현재 정의된 피스 목록으로 충분하며, 카탈로그 완성은 채보 제작 단계에서 이루어져도 된다.

## C. 구현 단계에서 결정해도 되는 사항

PRD에 포함할 필요가 없거나, 구현 시점에 결정해도 무방한 항목.

### 13. UI 화면 상세 디자인

`keybinding.md`의 UI 설계 요구사항(키 바인딩 설정 화면, 곡 선택 화면, 플레이 중 표시)은 구현/디자인 단계에서 결정.

### 14. 에디터의 모바일 지원 범위

`chart-editor.md`에 모바일 터치 조작이 정의되어 있으나, "개발자만 접근"하는 도구이므로 모바일 지원의 우선순위는 낮다. 구현 단계에서 결정.

### 15. 튜토리얼 Phase 5+

`tutorial.md`에서 이후 Phase는 TBD. 옵저버 모드 활용 튜토리얼, 고난이도 피스별 공략 가이드 등은 게임 출시 후 피드백에 따라 결정.

### 16. 성공 지표 (KPI)

PRD에 포함하면 좋지만, 비영리 무료 게임의 초기 단계에서는 다음 정도로 충분:
- DAU/MAU (활성 유저 수)
- 유저 리텐션 (1일/7일/30일)
- 평균 플레이 세션 길이
- 튜토리얼 완료율
- 난이도 등급별 플레이 분포

이 수치의 목표치는 출시 후 베이스라인 측정 이후에 설정하는 것이 현실적이다.

## D. 요약: PRD 작성 가능 여부 판단

### 반드시 선행 결정이 필요한 항목 (A 카테고리)

| # | 항목 | 상태 | 영향받는 PRD 섹션 |
|---|------|------|-------------------|
| 1 | 기술 스택 | **해결됨** (`tech-stack.md`) | 기술 요구사항, 아키텍처 |
| 2 | 차트 데이터 포맷 | **해결됨** (`tech-stack.md`) | 데이터 모델, API 설계 |
| 3 | MVP 스코프 확정 | **해결됨** (`mvp-scope.md`) | 기능 요구사항 전체 |
| 4 | 오디오 동기화 전략 | **해결됨** (`tech-stack.md`) | 기술 요구사항, 성능 |
| 5 | 타깃 유저 정의 | **해결됨** | 제품 개요, 커뮤니티 |

### 판단

**A 카테고리 5건 전부 해결됨. PRD 작성이 가능하다.**

22개 문서에 걸쳐 노트 시스템, 판정, 스코어링, 난이도, 키 바인딩, 차트 에디터, 에셋 수급 전략까지 상세히 정의되어 있으며, 대부분의 열린 질문은 이미 해결되었다 (`review-document-clarity.md` 참고). B 카테고리(6~12번)는 PRD에 "미정" 또는 잠정 값으로 표기하면 되며, C 카테고리(13~16번)는 구현 단계에서 결정해도 된다.
