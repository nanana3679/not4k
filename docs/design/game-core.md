# not4k — 게임 코어 설계

이 문서는 노트 시스템과 스코어링 이외의 게임 본체 동작 — 플랫폼, 인증, 랭킹, 입력 매칭, 스크롤, 오프셋, 게임 화면, 상태 흐름 — 을 정의한다.

---

## 플랫폼

**웹 게임**으로 개발한다. 브라우저에서 직접 플레이하며, 별도 설치가 필요 없다.

**웹을 선택한 이유**: not4k는 무료 게임이다. 무료 게임의 핵심 과제는 진입 장벽 최소화이며, 웹은 "URL 클릭 → 즉시 플레이"라는 가장 낮은 진입 장벽을 제공한다. 설치 과정이 유저 이탈의 주요 원인이 되는 점을 감안하면, 웹이 최적의 배포 방식이다. 부수적으로 Windows/Mac/Linux를 동시에 지원하며, 업데이트 배포가 즉각적이다.

### 온라인 전용

**오프라인 플레이는 지원하지 않는다.** 차트 데이터는 서버에서 스트리밍되며, 서버 연결이 필수이다.

**온라인 전용인 이유**: 첫째, 음원 저작권 비용 구조상 스트리밍이 유리하다. 저작권이 있는 음원을 사용할 경우, 유저에게 다운로드(복제)를 허용하면 복제권 라이선스 비용이 높아지지만, 스트리밍(전송)은 상대적으로 비용이 낮다. 향후 라이선스 음원을 도입할 가능성을 고려하면, 처음부터 스트리밍 기반으로 설계하는 것이 합리적이다. 둘째, 차트 추가·수정이 서버 측에서만 이루어지므로 배포가 단순하고, 랭킹과 플레이 기록도 서버 의존이므로 온라인 전용으로 통일하는 것이 구조적으로 일관된다. 오프라인 모드를 별도로 유지하면 로컬 캐싱, 동기화, 충돌 해결 등의 복잡도가 발생한다.

## 인증

### 소셜 로그인

**Google 소셜 로그인**을 제공한다.

**Google만 제공하는 이유**: 단일 제공자로 시작하여 구현 복잡도를 최소화한다. Google은 가장 범용적인 계정이며, 게임 유저 대부분이 보유하고 있다. 필요에 따라 추후 제공자를 추가할 수 있다.

### 비로그인 플레이

**로그인 없이도 플레이할 수 있다.** 단, 비로그인 상태에서는 다음 기능이 제한된다:

- 플레이 기록 저장 불가
- 랭킹 등록 불가
- 설정/오프셋 서버 동기화 불가

**비로그인 플레이를 허용하는 이유**: 웹 게임의 진입 장벽 최소화 원칙과 일치한다. "계정 만들기"가 첫 플레이 전에 강제되면 이탈률이 급증한다. 유저가 먼저 게임을 체험하고, 기록을 남기고 싶을 때 자발적으로 로그인하는 흐름이 전환율 측면에서 유리하다.

## 랭킹

### 차트별 랭킹

**차트(곡 + 난이도) 단위로 달성률 순위**를 제공한다. 로그인한 유저의 플레이 기록 중 최고 달성률이 차트별 랭킹에 등록된다.

**차트별 랭킹을 선택한 이유**: 리듬게임의 경쟁은 특정 차트에서의 성적 비교가 핵심이다. 전체 순위(볼포스류)는 난이도 책정이 정확해야 의미가 있으므로 추후 도입하고, 차트별 랭킹은 난이도 책정과 무관하게 즉시 동작한다. 유저가 "이 곡에서 내가 몇 등인지" 확인하는 것이 가장 직접적인 동기 부여다.

## 입력-노트 매칭

### 매칭 규칙

유저가 키를 누르면, 해당 레인에서 **가장 이른(earliest) 노트**에 매칭된다. 판정 윈도우(Bad ±160ms) 안에 복수의 노트가 존재하면, 시간순으로 가장 앞에 있는 노트가 입력을 흡수한다.

**가장 이른 노트 매칭을 선택한 이유**: "가장 가까운 노트" 매칭은 이른 입력이 뒤의 노트에 흡수되어 앞의 노트가 Miss 처리되는 직관적이지 않은 상황을 만든다. 가장 이른 노트 매칭은 노트의 판정 순서가 항상 배치 순서와 일치하므로, 유저가 결과를 예측하기 쉽다. osu!mania 등 대부분의 리듬게임이 이 방식을 사용한다.

### 빈 레인 입력

시각적으로 비어 보이는 레인에서 키를 눌렀을 때, 해당 레인에 Bad 윈도우(±160ms) 안에 노트가 존재하면 **Bad 판정**이 발생한다. 윈도우 안에 노트가 없으면 입력은 무시된다.

**빈 레인에서 Bad를 부여하는 이유**: 노트가 화면에 보이지 않더라도 Bad 윈도우에 이미 진입했다면, 그 입력은 해당 노트에 대한 시도로 간주하는 것이 자연스럽다. 입력을 완전히 무시하면, 유저가 무의미한 키 입력으로 아무런 페널티 없이 리듬을 탐색하는 것이 가능해진다. Bad 판정은 콤보를 파괴하고 풀콤보를 불허하므로, 정확한 입력에 대한 동기를 유지한다.

## 스크롤 속도

### BPM 독립 절대 속도

스크롤 속도(노트 낙하 속도)는 **BPM과 무관한 절대 속도**이며, 유저가 자유롭게 조절할 수 있다.

- 곡의 BPM이 변해도 스크롤 속도는 일정하다.
- BPM 변화 구간에서 노트 간격이 변하지 않으므로, 유저의 읽기 리듬이 깨지지 않는다.

**BPM 독립 속도를 선택한 이유**: BPM 연동 속도(IIDX의 Hi-Speed)는 BPM이 변하는 곡에서 소프란(soflan) 현상을 만든다. 유저가 BPM 변화 구간을 암기해야 하며, 이것은 리듬 감각이 아닌 암기력을 테스트하는 요소다. not4k는 가변 손배치와 차트 인식이라는 핵심 인지 부하에 집중해야 하므로, 스크롤 속도를 BPM에서 분리하여 불필요한 인지 부하를 제거한다. osu!mania, Arcaea 등 최근 리듬게임은 대부분 이 방식을 채택하고 있다.

## 오프셋

### 오디오 오프셋 (Audio Offset)

오디오 출력 지연을 보정하는 값. 음악 재생 타이밍을 전체적으로 이동시킨다. 출력 장치(스피커, 헤드폰, 블루투스 등)에 따라 달라지는 오디오 지연을 유저가 보정할 수 있다.

### 입력 오프셋 (Input Offset)

입력 장치 지연을 보정하는 값. 유저의 키 입력 타이밍을 전체적으로 이동시킨다. 키보드의 입력 지연이나 유저 개인의 체감 타이밍 차이를 보정한다.

### 두 오프셋을 분리한 이유

오디오 지연과 입력 지연은 원인이 다르다. 블루투스 이어폰을 사용하면 오디오 지연만 크게 증가하고, 무선 키보드를 사용하면 입력 지연만 증가한다. 단일 오프셋으로는 이 두 가지를 독립적으로 보정할 수 없다. 유저가 출력 장치를 교체했을 때 오디오 오프셋만 재조정하면 되고, 입력 오프셋은 유지할 수 있다.

### 오프셋 캘리브레이션

전용 캘리브레이션 화면을 제공한다. 유저가 설정 화면에서 진입하여 오디오/입력 오프셋을 각각 측정·조정할 수 있다.

**전용 화면을 제공하는 이유**: 오프셋 조정은 수치를 직접 입력하는 것보다 실제 음원에 맞춰 탭하고 편차를 측정하는 방식이 정확하고 직관적이다. 리듬게임 초보자에게 "오프셋을 ms 단위로 조정하세요"는 의미가 없으므로, 가이드된 측정 과정을 제공해야 한다.

## 게이지

**게이지는 존재하지 않는다.** 모든 플레이는 무조건 완주이며, 게임 오버가 없다.

**게이지를 제거한 이유**: not4k의 핵심 경험은 가변 손배치의 학습이다. 게이지에 의한 중도 실패는 이 학습 과정을 방해한다. 유저가 자신의 실력보다 어려운 차트에 도전하더라도 끝까지 플레이하며 판정 분포를 확인하고, 어떤 구간에서 실패했는지를 파악할 수 있어야 한다. 성적 평가는 달성률과 랭크(`scoring.md` 참고)로 충분하다.

---

## 미러 옵션

레인 배치를 좌우 반전하는 미러 옵션을 제공한다. 1-2-3-4 레인이 4-3-2-1로 뒤집힌다.

**미러를 제공하는 이유**: 왼손잡이 유저를 위한 접근성 보장이 주 목적이다. 키보드 리듬게임은 왼손이 낮은 레인, 오른손이 높은 레인을 담당하는 것이 관례인데, 왼손잡이에게는 주력 손이 단순한 파트를 담당하게 되어 불리하다. 미러 옵션으로 레인을 반전하면 오른손잡이와 동등한 조건이 된다. 부수적으로, 차트의 변주 효과도 제공한다.

## 게임패드

**게임패드는 지원하지 않는다.**

**게임패드를 지원하지 않는 이유**: not4k의 핵심인 "레인당 다중키 바인딩"은 한 손으로 같은 레인의 2개 이상의 키를 빠르게 교대하는 것을 전제로 한다. 게임패드의 버튼 배치는 이 동작에 적합하지 않다 — 엄지로 조작하는 페이스 버튼은 트릴 교대가 극도로 불편하고, 트리거/범퍼는 키 수가 부족하다. 게임패드를 지원하면 "게임패드로도 플레이할 수 있다"는 기대가 생기지만, 실제로는 핵심 경험을 제대로 전달할 수 없으므로 지원하지 않는다.

---

## 차트 에디터

**차트 에디터는 별도 애플리케이션으로 분리하며, 개발자만 접근한다.**

**별도 앱으로 분리하는 이유**: 차트 에디터는 게임 플레이와 완전히 다른 UI/UX를 필요로 한다 — 타임라인 편집, 노트 배치, 미리보기 등은 게임 클라이언트의 구조와 양립하기 어렵다. 분리함으로써 게임 클라이언트의 복잡도를 낮추고, 에디터의 기능을 게임의 릴리즈 주기에 구속받지 않고 독립적으로 개발할 수 있다. 유저 공개 에디터(커뮤니티 채보)는 현재 계획에 없으므로, 개발자 전용 도구로 충분하다.

차트 에디터의 상세 설계는 `chart-editor.md`를 참고한다.

## 게임 화면

### 필수 화면

| 화면 | 설명 |
|------|------|
| **타이틀** | 게임 시작 화면 |
| **첫 기동 설정** | 최초 실행 시 프리셋(키 바인딩) 설정 |
| **곡 선택** | 차트 선택, 옵션 설정, 곡 프리뷰 재생 |
| **플레이** | 노트 낙하, 입력, 판정이 이루어지는 핵심 화면 |
| **결과** | 달성률, 랭크, 판정 분포(Perfect/Great/Good/Good◇/Bad/Miss) 표시 |

### 기능 화면

| 화면 | 설명 |
|------|------|
| **설정** | 스크롤 속도, 키 바인딩, 오디오/비주얼 설정 등 |
| **오프셋 캘리브레이션** | 오디오/입력 오프셋 측정·조정 전용 화면 |
| **튜토리얼** | 노트 연주법과 판정 방식의 단계적 학습. 상세는 `tutorial.md` 참고 |
| **옵저버 모드** | 입력 없이 차트를 관찰하는 모드. 상세는 `observer-mode.md` 참고 |
| **스페셜 땡스** | 음원·이미지 등 에셋 제공자, 라이선스 표기, 기여자 크레딧을 표시하는 화면. 타이틀 화면에서 진입 |

### 곡 선택 화면의 곡 프리뷰

곡 선택 화면에서 커서가 위치한 곡의 **프리뷰 음원이 자동 재생**된다. 유저가 곡을 선택하기 전에 어떤 곡인지 들을 수 있어야 한다. 대부분의 리듬게임(osu!, DJMAX, SDVX 등)이 이 기능을 제공하며, 리듬게임 유저에게 기대되는 기본 기능이다.

**프리뷰 음원은 본편 음원과 별개의 오디오 파일**이다. 차트 메타데이터(`chart-editor.md`)에 프리뷰 전용 음원파일 필드를 두어, 채보 제작자가 곡의 하이라이트 구간을 별도로 편집하여 제공한다. 본편 음원에서 시작/종료 지점을 계산하는 방식이 아니므로, 페이드 인/아웃, 길이, 편집 등을 프리뷰 파일 자체에서 자유롭게 제어할 수 있다.

## 게임 상태 흐름

### 기본 흐름

```
타이틀 → [첫 기동 감지] → 프리셋 설정 → 튜토리얼 → 곡 선택
타이틀 → [이후 기동] → 곡 선택
```

첫 기동 시 프리셋 설정과 튜토리얼을 거치는 이유: not4k는 "레인당 다중키 바인딩"이라는 기존 리듬게임에 없는 개념을 전제로 한다. 키 바인딩 없이 플레이할 수 없으므로 프리셋 설정이 필수이고, 트릴 노트와 더블 노트의 존재를 인지하지 못하면 첫 플레이에서 혼란이 발생하므로 튜토리얼 안내가 필요하다.

### 플레이 흐름

```
곡 선택 → 플레이 → 결과 → 곡 선택
곡 선택 → 옵저버 모드 → 곡 선택
```

### 일시정지

플레이와 옵저버 모드에서 **일시정지가 가능**하다. 일시정지 중에는 **노트 출력부(판정선, 낙하 노트 등)가 가려진다.**

**노트 출력부를 가리는 이유**: 일시정지 중에 다가오는 노트를 미리 읽고 손배치를 계획하면, 실시간 인지 능력이 아닌 암기와 준비로 성적을 올릴 수 있다. 이는 플레이의 공정성을 해치므로, 일시정지 중에는 노트를 볼 수 없게 한다.

일시정지에서는 **재개** 또는 **중도 포기**를 선택할 수 있다.

### 중도 포기

플레이 도중 중도 포기하면 **결과 화면을 거치지 않고 곡 선택 화면으로 돌아간다.** 중도 포기한 플레이는 기록되지 않는다.

**결과 화면을 표시하지 않는 이유**: 중도 포기는 "이 차트를 더 이상 진행할 의사가 없다"는 의미이다. 완주하지 않은 플레이의 달성률과 랭크는 유저에게 유의미한 정보가 아니며, 불완전한 결과를 보여주는 것은 부정적 경험을 강화할 뿐이다. 게이지가 없어 게임 오버가 존재하지 않으므로, 중도 포기는 순전히 유저의 자발적 선택이다.

### 설정 접근

```
곡 선택 → 설정 → 곡 선택
설정 → 오프셋 캘리브레이션 → 설정
곡 선택 → 튜토리얼 (재진입) → 곡 선택
```

튜토리얼은 첫 기동 이후에도 곡 선택 화면에서 재진입할 수 있다. 유저가 특정 메커니즘을 복습하고 싶을 때 접근할 수 있어야 한다.

### 스페셜 땡스

```
타이틀 → 스페셜 땡스 → 타이틀
```

**타이틀에서 접근하는 이유**: 스페셜 땡스는 게임 플레이와 무관한 크레딧 정보이므로, 플레이 흐름(곡 선택 → 플레이 → 결과) 안에 배치할 필요가 없다. 타이틀 화면에 배치하여 관심 있는 유저만 자발적으로 접근하도록 한다.

**스페셜 땡스에 표시하는 내용**: 음원 제공자(아티스트), 이미지·에셋 제공자, 오픈소스 라이브러리 라이선스(`project-assets.md`의 저작권 표기 의무 이행), 기타 기여자 크레딧. 라이선스에 의해 표기가 의무인 항목은 반드시 이 화면에 포함한다.
