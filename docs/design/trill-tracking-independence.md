# not4k — 다중 레인 동시 트릴 교대 추적의 독립성

> 본 문서는 2개 이상의 레인에서 동시에 트릴이 진행되는 경우의 교대 추적 규칙을 정의한다.

---

## 문제 정의

not4k에서 트릴 노트는 "직전에 입력한 키와 다른 키를 눌러야 한다"는 교대 규칙을 가진다. 이 "직전 키"를 추적하는 상태(이하 **교대 추적 상태**)가 레인마다 독립적인지, 또는 전역적으로 공유되는지에 따라 판정 결과가 달라진다.

### 발생 시나리오

트릴 앵커 피스(PP-002, PLv.7)에서 한 레인이 트릴을 전담하고 반대 손이 나머지 레인을 처리하는 구조가 기본이다. 그러나 채보 디자이너가 **2개 이상의 레인에 동시 트릴**을 배치하는 것은 규칙상 가능하다.

예시: L1과 L4에 동시 트릴 앵커

```
16bit (4레인 수직 표기)
L1  L2  L3  L4
{   .   .   {
t   .   o   t
t   o   .   t
t   .   o   t
t   o   .   t
}   .   .   }
```

이 배치에서 L1의 교대 추적과 L4의 교대 추적이 독립적이지 않다면, L1에서의 입력이 L4의 교대 판정에 영향을 미치는 비직관적인 결과가 발생할 수 있다.

---

## 설계 결정: 레인별 독립 추적

### 규칙

**각 레인의 트릴 교대 추적은 완전히 독립적이다.**

- 각 레인은 자체적인 "직전 입력 키" 상태를 가진다.
- L1에서의 키 입력은 L1의 교대 추적 상태만 갱신한다.
- L4에서의 키 입력은 L4의 교대 추적 상태만 갱신한다.
- 어떤 레인의 트릴 교대 성공/실패가 다른 레인의 판정에 영향을 주지 않는다.

### 의사코드

```
// 레인별 독립 교대 추적 상태
lane_last_key: Map<LaneId, KeyId | null>

function on_trill_note_input(lane: LaneId, key: KeyId):
    last = lane_last_key[lane]
    if last == null or last != key:
        // 교대 성공: 정상 판정 (타이밍 기반)
        judge = evaluate_timing(...)
    else:
        // 교대 실패: Good◇ 고정
        judge = Good_Diamond
    lane_last_key[lane] = key
```

## 독립 추적이 필요한 이유

### 1. 물리적 독립성

2개 레인의 동시 트릴은 양손이 각각 담당한다. L1은 왼손, L4는 오른손이 처리하므로, 각 손의 교대 상태는 물리적으로 완전히 독립적이다. 전역 추적은 이 물리적 현실과 모순된다.

### 2. 인지적 독립성

유저는 각 레인의 트릴을 독립적인 작업으로 인식한다. "L1에서 A→B→A→B, L4에서 C→D→C→D"를 각각 독립적으로 실행하며, 한쪽의 실수가 다른 쪽에 전파되지 않아야 한다.

### 3. 설계 일관성

not4k의 모든 판정은 레인 단위로 동작한다:

- 입력-노트 매칭: 레인별로 "가장 이른 노트"에 매칭 (`game-core.md`)
- 롱노트 유지 판정: 레인별로 독립 판정 (`note-system.md`)
- 더블 롱노트 바디: "독립된 싱글 바디 × 2로 병렬 판정" (`scoring.md`)

트릴 교대 추적만 전역적으로 동작한다면 이 일관성이 깨진다.

### 4. 채보 디자인의 자유도

독립 추적이 보장되어야 채보 디자이너가 다중 레인 트릴을 안심하고 배치할 수 있다. 전역 추적은 "L1에서 키 A를 눌렀으므로 L4에서도 A 외의 키를 눌러야 한다"는 불합리한 제약을 만든다 — L1과 L4는 서로 다른 키가 바인딩되어 있으므로 이 제약 자체가 성립하지 않는다.

## 엣지 케이스

### 같은 레인에서 트릴 구간이 연속되는 경우

```
16bit
L2
{
t
t
}
.
{
t
t
}
```

첫 번째 트릴 구간의 마지막 입력 키가 두 번째 트릴 구간의 첫 번째 노트에 영향을 미치는가?

**결정**: 트릴 구간이 끝나면 해당 레인의 교대 추적 상태가 **초기화(null)**된다. 새 트릴 구간의 첫 번째 노트는 항상 교대 성공으로 판정된다. 이유: 트릴 구간 사이의 빈 틱에서 유저가 손배치를 재정비할 수 있으므로, 이전 구간의 상태를 계승하면 유저의 의도와 맞지 않는 판정이 발생할 수 있다.

### 트릴 구간의 첫 번째 노트

트릴 구간의 첫 번째 트릴 노트는 "직전 키"가 존재하지 않으므로(null), **어떤 키를 눌러도 교대 성공**이다. 이는 트릴 구간 진입의 자연스러움을 보장한다.

### 3키 이상 바인딩에서의 다중 레인 트릴

레인당 3키(A, B, C) 바인딩에서 교대 규칙은 "직전 키와 다르면 성공"이다. `A → B → C → A`는 모두 교대 성공이며, `A → B → B → C`에서 세 번째 B만 교대 실패이다. 이 규칙은 다중 레인 환경에서도 레인별로 독립 적용된다.

## 구현 시 주의사항

### 상태 관리

- 교대 추적 상태는 `Map<LaneId, KeyId | null>`로 관리한다.
- 트릴 구간 시작 시 해당 레인의 상태를 null로 초기화한다.
- 트릴 구간 종료 시 해당 레인의 상태를 null로 초기화한다.
- 트릴 노트 입력 시 해당 레인의 상태만 갱신한다.

### 디버깅

- 다중 레인 트릴 테스트 시, 각 레인의 교대 상태가 독립적으로 변하는지를 시각적으로 확인할 수 있는 디버그 UI가 유용하다.
- 옵저버 모드에서 교대 추적 상태의 변화를 레인별로 표시하는 기능을 고려할 수 있다.
