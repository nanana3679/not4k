# not4k — 전체 문서 및 PRD 종합 리뷰

> 26개 문서(README, PRD, 설계 16개, 리서치 6개, AGENTS.md)를 전수 검토한 결과이다.

---

## 총평

not4k의 설계 문서는 기획 단계의 프로젝트로서 **매우 높은 완성도**를 가지고 있다. "레인당 다중키 바인딩"이라는 단일 핵심 메커니즘에서 파생되는 모든 설계 결정이 일관된 논리 구조를 유지하고 있으며, 각 결정의 근거가 기존 리듬게임의 역기획 분석에 기반하여 탄탄하다. 이전 리뷰(`review-document-clarity.md`)에서 지적된 19개 항목 중 대부분이 해결되었고, 용어 통일과 문서 간 교차 참조도 양호한 수준이다.

아래에서는 **남아있는 문제**, **새로 발견한 불일치**, **설계상 보완이 필요한 영역**, **구현 진입 시 우선 확정이 필요한 미정 항목**을 순서대로 정리한다.

## A. 문서 간 불일치 및 오류

### A-1. PP 번호 체계의 불연속

`piece-definition.md`의 피스 카탈로그에 PP-002가 삭제되었으나, 번호 체계가 PP-001, PP-003~PP-011로 남아 있다. PP-002가 의도적으로 결번인지 설명이 없으면, 이후 피스를 추가할 때 혼란이 생길 수 있다.

- **권장**: PP-002를 "결번(삭제됨)" 으로 명시하거나, 전체 번호를 재배정

### A-2. chart-design.md의 PLv.11 참조

`chart-design.md` 174행에 "초급(Lv.4~6) 채보에 가변 분할 전환(PLv.11)이 등장하면 안 된다"고 기술되어 있으나, `piece-definition.md`에서 가변 분할 전환(PP-007)의 PLv.은 **14**이다. PLv.11은 어떤 피스에도 할당되어 있지 않다.

- **권장**: `chart-design.md`의 PLv.11을 PLv.14로 수정

### A-3. 키 바인딩 최소 요구사항의 불일치

- `keybinding.md` 244행: "유저는 각 레인에 **2개 이상**의 키를 바인딩**해야** 한다"
- `prd.md` 228행: "레인당 **2키 이상** 바인딩 필수 (최소 8키, 최대 10키+)"

그러나 `keybinding.md`의 입문(Lv.1~3) 설명에서는 "중지와 검지 **4키를 주로 활용**"하며 "주로 W만", "주로 E만" 등 사실상 **1키만 사용하는 것이 기본 플레이**임을 서술한다. 시스템이 레인당 2키 바인딩을 **강제**하는 것인지, 아니면 2키를 **설정하되 1키만 사용하는 것을 허용**하는 것인지 명확하지 않다.

- **권장**: "8키 바인딩을 기본으로 설정하되, 입문에서는 4키를 주로 사용한다"는 현 서술이 맞다면, "바인딩은 레인당 2키 이상 필수이나, 모든 키를 활용할 의무는 없다"는 점을 명시

### A-4. overview.md 피스 테이블의 누락

`overview.md`의 피스 요약 테이블에는 릴리즈탭(PLv.4)과 홀드 중 탭(PLv.7) 2개만 기재되어 있다. 트릴 앵커(PLv.7), 홀드 트릴(PLv.7), 비트 복합 트릴(PLv.10), 건너가기(PLv.14) 등 주요 피스가 누락되어 있어 진입 문서로서의 개요 기능이 부족하다.

- **권장**: 주요 피스 6~8개 정도로 테이블을 확장하거나, "대표 2개만 소개하며 전체 목록은 `piece-definition.md` 참고"라는 안내를 추가

### A-5. chart-editor.md의 편집 모드 아이콘 표기 오류

`project-assets.md` 117행에서 편집 모드 아이콘을 "Create/Move/Select/Delete 모드 아이콘 — 4개 모드"로 기술하고 있으나, `chart-editor.md`에서는 편집 모드를 Create/Select/Delete **3개**로 정의하며, "이동(Move)은 Select 모드의 기능으로 통합"한다고 명시한다.

- **권장**: `project-assets.md`에서 "Create/Move/Select/Delete" → "Create/Select/Delete"로 수정 (3개 모드)

### A-6. 릴리즈 판정 상향 규칙의 문서 간 산포

릴리즈 판정 시 Good 이상이 Perfect로 상향되는 규칙이 `scoring.md`와 `glossary.md`에는 명시되어 있으나, `note-system.md`의 바디 판정 섹션에서는 "바디 판정의 결과는 Perfect 또는 Miss의 두 가지뿐이다"라고만 기술되어 있어 상향 메커니즘이 누락된 것처럼 보인다. `note-system.md`의 바디 판정 설명 자체가 상향 규칙을 이미 포함하고 있는지(Perfect 또는 Miss이므로 상향이 불필요한 구조인지) 불명확하다.

- **권장**: `note-system.md`에서 "바디 판정은 Perfect/Miss 이분법이므로 릴리즈 타이밍 정밀도를 요구하지 않는다"와 `glossary.md`의 "릴리즈 판정 상향: Good 이상 → Perfect" 간의 관계를 명확히 정리. 릴리즈 판정 상향이 바디 끝 판정에도 적용되는 것인지, 아니면 바디가 이미 이분법이므로 적용 대상이 아닌지 명시

## B. 설계상 미비점 및 엣지 케이스

### B-1. 더블 트릴 노트의 부재

트릴 노트(`t`)와 더블 노트(`D`)는 각각 독립적 노트 타입으로 정의되어 있으나, "더블 트릴 노트" (한 레인에서 서로 다른 2키를 동시에 누르되, 직전 쌍과 다른 키 조합을 요구하는 노트)에 대한 정의가 없다. 이것이 의도적으로 배제된 것인지, 아니면 미고려 사항인지 불분명하다.

- **권장**: 더블 트릴이 불필요하다면 의도적 배제 이유를 명시. 레인당 2키 기본 체계에서는 더블 노트가 이미 모든 키를 사용하므로 교대할 키가 남지 않아 물리적으로 불가능하다는 점을 `note-system.md`에 기술

### B-2. 트릴 구간 안에서 더블 노트 배치 불가의 근거

`note-system.md`에서 "트릴 구간에는 트릴 노트만 등장할 수 있다. 트릴 구간 안에 싱글 노트나 더블 노트가 배치될 수 없다"로 정의하나, 더블 노트가 배제되는 **이유**가 명시되지 않았다. 트릴 구간 중 임팩트가 필요한 시점에 더블 노트를 사용하고 싶은 채보 수요가 있을 수 있다.

- **권장**: 트릴 구간에서 더블 노트를 배제하는 근거(예: 교대 판정과 동시 입력 판정의 충돌, 인지 부하 과부하 방지 등)를 명시

### B-3. 동시 다중 레인 트릴의 판정 규칙

2개 이상의 레인에서 동시에 트릴이 진행되는 경우(예: L1과 L4 동시 트릴 앵커)의 판정 규칙이 명시되지 않았다. 각 레인의 교대 추적이 독립적이라면 문제가 없지만, 이를 명시해야 채보 디자이너의 혼란을 방지할 수 있다.

- **권장**: "각 레인의 트릴 교대 추적은 독립적이다" 한 줄 추가

### B-4. 유예 시간 12ms와 하드웨어 폴링 레이트의 관계

유예 시간(Grace Period) 12ms는 키 전환의 물리적 공백(5~15ms)을 기준으로 설정되었으나, `judgment-windows.md`에서 지적한 바와 같이 키보드 폴링 레이트(125Hz = 8ms, 1000Hz = 1ms)에 따라 실제 감지 가능한 최소 공백이 달라진다. 125Hz 키보드에서는 8ms 단위로만 입력을 감지하므로, 12ms 유예 시간이 실질적으로 8ms(1프레임)~16ms(2프레임) 범위로 동작할 수 있다.

- **권장**: 유예 시간의 측정 기준이 "게임 엔진의 타이밍 기준"인지 "하드웨어 입력 이벤트 타임스탬프 기준"인지 명시. 구현 시 고정밀 타임스탬프 기반이 아니면 125Hz 유저에게 불리할 수 있음을 인지

### B-5. 비로그인 유저의 설정 저장

`game-core.md`에서 비로그인 시 "설정/오프셋 서버 동기화 불가"로 기술하나, 비로그인 유저의 설정이 **로컬(브라우저 localStorage 등)에는 저장되는지** 여부가 미정의이다. 웹 게임에서 매번 키 바인딩과 오프셋을 재설정해야 한다면 비로그인 플레이의 진입 장벽이 크게 높아진다.

- **권장**: 비로그인 시에도 로컬 저장소에 설정을 보존하는 방안을 명시

### B-6. 곡 프리뷰의 시작 지점/길이 미정의

`game-core.md`에서 곡 선택 시 프리뷰 음원 자동 재생을 기술하나, 프리뷰의 시작 지점(사비 시작, 곡 시작, 채보 메타데이터에 지정 등)과 길이가 미정이다. `chart-editor.md`의 차트 메타데이터에도 프리뷰 관련 필드가 없다.

- **권장**: 차트 메타데이터에 `previewStart` (ms), `previewDuration` (ms) 필드를 추가하거나, 기본값 규칙을 정의

## C. PRD 검토

### C-1. 강점

- 문제 정의(3가지 구조적 한계)와 해법(레인당 다중키 바인딩)의 대응이 명확하고, 각각에 대한 근거 문서 링크가 완비되어 있다
- 제품 범위가 게임 클라이언트 / 차트 편집기 / 공유 도메인 / 백엔드 서버의 4개 패키지로 체계적으로 분류되어 있다
- 미정 사항이 별도 섹션(12절)에 솔직하게 나열되어 있어, 구현 팀이 어떤 결정이 남아 있는지 즉시 파악할 수 있다
- 성공 지표(13절)가 구체적이고 측정 가능하다

### C-2. 보완이 필요한 부분

#### 기술 스택 미정

PRD 10절(기술 요구사항)에서 "웹 게임"이라는 플랫폼만 결정되었고, 구체적인 기술 스택(프레임워크, 렌더링 엔진, 오디오 API 등)이 미정이다. 구현 단계 진입 전에 결정이 필요하다.

- 렌더링: Canvas 2D vs WebGL (Pixi.js, Three.js 등)
- 오디오: Web Audio API (저지연 필수, 오프셋 보정 필요)
- 상태 관리: 게임 루프 vs 프레임워크 기반
- 서버: REST vs WebSocket (차트 스트리밍 방식)
- 모노레포 도구: Turborepo, Nx 등

#### 비기능 요구사항 부재

성능 요구사항(목표 FPS, 오디오 레이턴시 상한, 최소 지원 브라우저 등)이 PRD에 없다. 리듬게임 특성상 입력 레이턴시와 오디오 동기화가 핵심이므로, 최소한 다음을 명시해야 한다:

- 목표 프레임레이트 (60fps? 120fps? 가변?)
- 허용 오디오 레이턴시 상한
- 최소 지원 브라우저 및 버전
- 동시 접속 용량 (서버 스트리밍 구조에 대한 규모 기대치)

#### 에러 처리 / 예외 시나리오 미정

- 플레이 중 네트워크 단절 시 동작 (온라인 전용이므로 중요)
- 음원 스트리밍 실패 시 동작
- 서버 점검 시 비로그인 유저의 플레이 가능 여부

#### 접근성 관련 추가 검토 필요

미러 옵션(왼손잡이)은 이미 고려되었으나, 다음이 미고려:

- 색각 이상 유저를 위한 노트 색상/형태 구분 보장 (더블 노트 강조 방식이 색상에만 의존할 경우 문제)
- 키보드 입력이 불가능한 장애 유저에 대한 접근성 정책

## D. 문서 구조 및 가독성

### D-1. README의 역할이 명확하다

README가 문서 인덱스 + 프로젝트 예상 구조 + 읽기 순서를 모두 포함하여 진입점으로서 잘 기능한다.

### D-2. 교차 참조의 일관성

대부분의 문서가 말미에 "관련 문서" 섹션을 가지고 있어 탐색이 용이하다. 다만 일부 문서에서 관련 문서 경로가 상대 경로(`overview.md`)로만 표기되어, 상위 디렉토리의 문서를 참조할 때 혼란이 생길 수 있다. (예: `rationale.md`에서 `../research/` 참조 시에는 올바르게 상대 경로를 사용하고 있으나, 같은 디렉토리 내의 문서는 파일명만 표기)

### D-3. review-document-clarity.md의 번호 건너뜀

review-document-clarity.md에서 항목 번호가 1~16, 17, 19, 18 순서로 기재되어 있다. 18과 19의 순서가 뒤바뀌어 있다.

- **권장**: 번호 순서 정리 (18 → 19가 아니라 17 → 18 → 19 순서로)

## E. 리서치 문서 검토

### E-1. 강점

- 6개 게임에 대한 역기획이 각각 독립적이면서도 not4k 설계 근거와 명확히 연결된다
- `judgment-windows.md`는 12개 이상 리듬게임의 판정 윈도우를 수집하여 `scoring.md`의 ±41ms 결정에 강력한 근거를 제공한다
- 각 게임의 고유 메커니즘(IIDX 스크래치, SDVX 노브, 츄니즘 슬라이더, maimai 원형 배치)에서 not4k로의 번역 경로가 일관되게 제시된다

### E-2. 보완 사항

- **osu!mania의 부재**: 타겟 유저(PRD 3절)에서 "기존 4K 유저 (osu!mania, DJMAX 등)"를 명시하고 있으나, osu!mania에 대한 독립 리서치 문서가 없다. not4k와 가장 직접적으로 비교될 PC 4키 리듬게임이므로, 별도 역기획 분석이 있으면 설계 근거가 더 강화될 것이다
- **Arcaea/Phigros 분석 부재**: `project-assets.md`에서 Arcaea와 Phigros의 성공 사례(무명 작곡가 곡으로 성공)를 인용하고 있으나, 이 게임들에 대한 리서치 문서는 없다. 에셋 전략의 근거로만 사용하려면 현 수준으로 충분하나, 게임플레이 참고까지 확장하려면 별도 문서가 필요할 수 있다

## G. 결론

not4k의 문서 체계는 기획 문서로서 사실상 완성 단계에 가깝다. 핵심 메커니즘, 노트 시스템, 판정 체계, 난이도 설계, 키 바인딩, 스코어링이 모두 상호 일관되게 정의되어 있으며, 6개 역기획 보고서가 각 설계 결정의 근거를 제공한다.

**구현 진입을 위해 가장 시급한 과제**는 다음 3가지이다:

1. **기술 스택 결정**: 렌더링, 오디오, 서버 통신 방식을 확정해야 모노레포 구조와 패키지 설계가 시작될 수 있다
2. **비기능 요구사항 정의**: 목표 FPS, 오디오 레이턴시 상한, 최소 지원 환경이 없으면 기술 선택의 기준이 불명확하다
3. **본 리뷰의 A절 불일치 수정**: PLv.11 → 14 오기, 편집 모드 4개 → 3개, PP-002 결번 처리 등 사실 오류를 수정해야 문서의 신뢰도가 유지된다

나머지 미정 사항(난이도명, 판정명, 비주얼, 볼포스류)은 구현 진행과 병행하여 결정해도 무방하다.
