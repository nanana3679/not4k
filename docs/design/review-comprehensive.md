# not4k — 전체 문서 및 PRD 종합 리뷰

> 26개 문서(README, PRD, 설계 16개, 리서치 6개, AGENTS.md)를 전수 검토한 결과이다.

---

## 총평

not4k의 설계 문서는 기획 단계의 프로젝트로서 **매우 높은 완성도**를 가지고 있다. "레인당 다중키 바인딩"이라는 단일 핵심 메커니즘에서 파생되는 모든 설계 결정이 일관된 논리 구조를 유지하고 있으며, 각 결정의 근거가 기존 리듬게임의 역기획 분석에 기반하여 탄탄하다. 이전 리뷰(`review-document-clarity.md`)에서 지적된 19개 항목 중 대부분이 해결되었고, 용어 통일과 문서 간 교차 참조도 양호한 수준이다.

아래에서는 **남아있는 문제**, **새로 발견한 불일치**, **설계상 보완이 필요한 영역**, **구현 진입 시 우선 확정이 필요한 미정 항목**을 순서대로 정리한다.

## A. 문서 간 불일치 및 오류

### ~~A-1. PP 번호 체계의 불연속~~ → **해결됨**

PP-002 결번을 제거하고 전체 번호를 재배정하였다 (PP-003→PP-002, PP-004→PP-003 등). 현재 피스 카탈로그는 PP-001~PP-010의 연속 번호 체계이다.

### ~~A-2. chart-design.md의 PLv.11 참조~~ → **해결됨**

가변 분할 전환, 가변 분할, 건너가기의 PLv.을 모두 **PLv.11**로 통일하였다. `chart-design.md`의 PLv.11 참조가 올바른 값이었으며, `piece-definition.md`의 PLv.14를 PLv.11로 수정하였다. 이 세 피스는 상급(Lv.11~13) 등급에 해당한다.

### A-3. 키 바인딩 최소 요구사항 → **명확화됨**

레인당 **최소 2키 바인딩이 필수**이다 (시스템이 강제). 기본 프리셋은 레인당 **3키 (총 12키)**로 변경되었다. 입문에서는 4키를 주로 활용하되, 바인딩 자체는 12키가 설정된 상태이다. "바인딩은 레인당 2키 이상 필수이나, 입문 단계에서는 모든 키를 활용할 의무가 없다"는 점이 `keybinding.md`에서 명확히 서술된다.

### ~~A-4. overview.md 피스 테이블의 누락~~ → **해결됨**

`overview.md`의 피스 요약 테이블을 전체 9개 피스(릴리즈탭, 싱글→더블 전환, 트릴 앵커, 홀드 중 탭, 홀드 트릴, 비트 복합 트릴, 건너가기, 가변 분할, 가변 분할 전환)로 확장하였다.

### ~~A-5. chart-editor.md의 편집 모드 아이콘 표기 오류~~ → **해결됨**

`project-assets.md`에서 "Create/Move/Select/Delete — 4개 모드"를 "Create/Select/Delete — 3개 모드 (이동은 Select에 통합)"로 수정하였다.

### ~~A-6. 릴리즈 판정 상향 규칙의 문서 간 산포~~ → **해결됨**

바디 판정을 Perfect/Miss 이분법에서 **Perfect/Bad/Miss 삼분법**으로 변경하였다. Good 이상은 Perfect로 상향되고, Bad는 그대로 Bad, 바디 유지 실패는 Miss이다. `note-system.md`, `scoring.md`, `glossary.md`, `prd.md`를 모두 통일하여 릴리즈 판정 상향 규칙과 바디 판정 구조의 관계가 명확해졌다.

## B. 설계상 미비점 및 엣지 케이스

### ~~B-1. 더블 트릴 노트의 부재~~ → **해결됨**

더블 트릴 노트는 의도적으로 정의하지 않는다. `note-system.md`에 "더블 트릴 노트가 존재하지 않는 이유" 섹션을 추가하였다. 핵심 근거: 더블 노트(2키 동시)에서 미세한 입력 시간차로 인해 "직전 키"가 비결정적이 되어, 다음 트릴 노트의 교대 판정을 신뢰할 수 없다. 레인당 2키 기본 체계에서는 더블 노트가 이미 모든 키를 사용하므로 교대할 키가 남지 않는다는 물리적 제약도 존재한다.

### B-2. 트릴 구간 안에서 더블 노트 배치 불가의 근거

`note-system.md`에서 "트릴 구간에는 트릴 노트만 등장할 수 있다. 트릴 구간 안에 싱글 노트나 더블 노트가 배치될 수 없다"로 정의하나, 더블 노트가 배제되는 **이유**가 명시되지 않았다. 트릴 구간 중 임팩트가 필요한 시점에 더블 노트를 사용하고 싶은 채보 수요가 있을 수 있다.

- **권장**: 트릴 구간에서 더블 노트를 배제하는 근거(예: 교대 판정과 동시 입력 판정의 충돌, 인지 부하 과부하 방지 등)를 명시

### ~~B-3. 동시 다중 레인 트릴의 판정 규칙~~ → **해결됨**

각 레인의 트릴 교대 추적은 완전히 독립적임을 명시하였다. `note-system.md`에 "다중 레인 트릴의 독립성" 섹션을 추가하고, 상세 분석 보고서 `trill-tracking-independence.md`를 작성하였다. 트릴 구간 시작/종료 시 교대 추적 상태 초기화, 엣지 케이스(연속 트릴 구간, 첫 번째 노트), 구현 의사코드를 포함한다.

### ~~B-4. 유예 시간 12ms와 하드웨어 폴링 레이트의 관계~~ → **해결됨**

유예 시간 12ms를 유지하되, KeyboardEvent의 고해상도 타임스탬프(`DOMHighResTimeStamp`) 기반으로 측정하도록 확정하였다. `note-system.md`에 측정 기준을 명시하고, 상세 분석 보고서 `grace-period-polling-rate.md`를 작성하였다. 125Hz 키보드에서의 양자화 문제, rAF 프레임 주기와의 분리, 동적 보정의 기각 근거를 포함한다.

### B-5. 비로그인 유저의 설정 저장

`game-core.md`에서 비로그인 시 "설정/오프셋 서버 동기화 불가"로 기술하나, 비로그인 유저의 설정이 **로컬(브라우저 localStorage 등)에는 저장되는지** 여부가 미정의이다. 웹 게임에서 매번 키 바인딩과 오프셋을 재설정해야 한다면 비로그인 플레이의 진입 장벽이 크게 높아진다.

- **권장**: 비로그인 시에도 로컬 저장소에 설정을 보존하는 방안을 명시

### ~~B-6. 곡 프리뷰의 시작 지점/길이 미정의~~ → **해결됨**

프리뷰 음원은 본편 음원과 **별개의 오디오 파일**로 확정하였다. `chart-editor.md`의 차트 메타데이터에 "프리뷰 음원파일" 필드를 추가하고, `game-core.md`에 별개 파일 방식의 근거를 명시하였다. 채보 제작자가 하이라이트 구간을 별도 편집하여 제공하므로, 시작 지점/길이/페이드를 프리뷰 파일 자체에서 제어한다.

## C. PRD 검토

### C-1. 강점

- 문제 정의(3가지 구조적 한계)와 해법(레인당 다중키 바인딩)의 대응이 명확하고, 각각에 대한 근거 문서 링크가 완비되어 있다
- 제품 범위가 게임 클라이언트 / 차트 편집기 / 공유 도메인 / 백엔드 서버의 4개 패키지로 체계적으로 분류되어 있다
- 미정 사항이 별도 섹션(12절)에 솔직하게 나열되어 있어, 구현 팀이 어떤 결정이 남아 있는지 즉시 파악할 수 있다
- 성공 지표(13절)가 구체적이고 측정 가능하다

### C-2. 보완이 필요한 부분

#### 기술 스택 미정

PRD 10절(기술 요구사항)에서 "웹 게임"이라는 플랫폼만 결정되었고, 구체적인 기술 스택(프레임워크, 렌더링 엔진, 오디오 API 등)이 미정이다. 구현 단계 진입 전에 결정이 필요하다.

- 렌더링: Canvas 2D vs WebGL (Pixi.js, Three.js 등)
- 오디오: Web Audio API (저지연 필수, 오프셋 보정 필요)
- 상태 관리: 게임 루프 vs 프레임워크 기반
- 서버: REST vs WebSocket (차트 스트리밍 방식)
- 모노레포 도구: Turborepo, Nx 등

#### 비기능 요구사항 부재

성능 요구사항(목표 FPS, 오디오 레이턴시 상한, 최소 지원 브라우저 등)이 PRD에 없다. 리듬게임 특성상 입력 레이턴시와 오디오 동기화가 핵심이므로, 최소한 다음을 명시해야 한다:

- 목표 프레임레이트 (60fps? 120fps? 가변?)
- 허용 오디오 레이턴시 상한
- 최소 지원 브라우저 및 버전
- 동시 접속 용량 (서버 스트리밍 구조에 대한 규모 기대치)

#### 에러 처리 / 예외 시나리오 미정

- 플레이 중 네트워크 단절 시 동작 (온라인 전용이므로 중요)
- 음원 스트리밍 실패 시 동작
- 서버 점검 시 비로그인 유저의 플레이 가능 여부

#### 접근성 관련 추가 검토 필요

미러 옵션(왼손잡이)은 이미 고려되었으나, 다음이 미고려:

- 색각 이상 유저를 위한 노트 색상/형태 구분 보장 (더블 노트 강조 방식이 색상에만 의존할 경우 문제)
- 키보드 입력이 불가능한 장애 유저에 대한 접근성 정책

### D-3. review-document-clarity.md의 번호 건너뜀

review-document-clarity.md에서 항목 번호가 1~16, 17, 19, 18 순서로 기재되어 있다. 18과 19의 순서가 뒤바뀌어 있다.

## E. 리서치 문서 검토

### E-1. 강점

- 6개 게임에 대한 역기획이 각각 독립적이면서도 not4k 설계 근거와 명확히 연결된다
- `judgment-windows.md`는 12개 이상 리듬게임의 판정 윈도우를 수집하여 `scoring.md`의 ±41ms 결정에 강력한 근거를 제공한다
- 각 게임의 고유 메커니즘(IIDX 스크래치, SDVX 노브, 츄니즘 슬라이더, maimai 원형 배치)에서 not4k로의 번역 경로가 일관되게 제시된다

### E-2. 보완 사항

- **osu!mania의 부재**: 타겟 유저(PRD 3절)에서 "기존 4K 유저 (osu!mania, DJMAX 등)"를 명시하고 있으나, osu!mania에 대한 독립 리서치 문서가 없다. not4k와 가장 직접적으로 비교될 PC 4키 리듬게임이므로, 별도 역기획 분석이 있으면 설계 근거가 더 강화될 것이다
- **Arcaea/Phigros 분석 부재**: `project-assets.md`에서 Arcaea와 Phigros의 성공 사례(무명 작곡가 곡으로 성공)를 인용하고 있으나, 이 게임들에 대한 리서치 문서는 없다. 에셋 전략의 근거로만 사용하려면 현 수준으로 충분하나, 게임플레이 참고까지 확장하려면 별도 문서가 필요할 수 있다

## G. 결론

not4k의 문서 체계는 기획 문서로서 사실상 완성 단계에 가깝다. 핵심 메커니즘, 노트 시스템, 판정 체계, 난이도 설계, 키 바인딩, 스코어링이 모두 상호 일관되게 정의되어 있으며, 6개 역기획 보고서가 각 설계 결정의 근거를 제공한다.

**구현 진입을 위해 가장 시급한 과제**는 다음 3가지이다:

1. **기술 스택 결정**: 렌더링, 오디오, 서버 통신 방식을 확정해야 모노레포 구조와 패키지 설계가 시작될 수 있다
2. **비기능 요구사항 정의**: 목표 FPS, 오디오 레이턴시 상한, 최소 지원 환경이 없으면 기술 선택의 기준이 불명확하다
3. **~~본 리뷰의 A절 불일치 수정~~** → **해결됨**: PP-002 결번 제거(번호 재배정), PLv.14→11 통일, overview.md 피스 테이블 확장, 바디 판정 삼분법 통일, 더블 트릴 불가 사유 명시 등이 완료되었다

나머지 미정 사항(난이도명, 판정명, 비주얼, 볼포스류)은 구현 진행과 병행하여 결정해도 무방하다.
