# not4k — MVP 스코프

> 핵심 질문: **"가변 손배치 4레인 리듬게임이 재미있는가?"**

이 문서는 위 질문에 답하기 위한 최소 기능 범위(Phase A)와, 공개 빌드를 위한 기능 범위(Phase B)를 정의한다. `project-assets.md`의 에셋 단계와 대응하되, **기능적 스코프**에 집중한다.

---

## Phase A: 프로토타입 (핵심 메커니즘 검증)

### 목표

내부 테스터에게 코어 게임플레이를 체험시키고, 다음을 검증한다:

- 트릴/더블 노트가 재미있는가?
- AudioContext 타이밍이 정확한가?
- 가변 손배치가 기존 4키와 체감 차이가 있는가?

### 포함 기능

#### 게임 (`apps/game`)

| 카테고리 | 기능 | 근거 |
|---------|------|------|
| **화면** | 타이틀(최소), 프리셋 설정, 곡 선택(기본 리스트), 플레이, 결과 | 게임 루프의 최소 단위 |
| **게임플레이** | 4레인 노트 낙하, 싱글/롱/트릴/더블 전 노트 타입, 판정(Perfect~Miss+TGood) | 모든 노트 타입이 Lv.1부터 등장하므로 전부 필요 (`overview.md`) |
| **오디오** | AudioContext 마스터 클럭, AudioBuffer 전체 로드 | 판정 정확도 검증이 핵심 |
| **키 바인딩** | 프리셋 선택(넘패/TKL) + 레인별 커스텀 바인딩 | 키 바인딩 없으면 플레이 불가 |
| **스크롤 속도** | BPM 독립 절대 속도 조절 | 리듬게임 기본 기능 |
| **결과** | 달성률, 랭크(SSS~F), 판정 분포(TGood 별도 행) | 피드백 루프 필수 |
| **인증** | 비로그인(게스트) 플레이만 | 진입 장벽 최소화, 인증은 Phase B |
| **콘텐츠** | CC0/CC BY 곡 3~5곡 | 메커니즘 검증에 충분한 최소 곡수 |
| **인프라** | Supabase Storage에서 차트 JSON + 음원 fetch | 온라인 전용 구조 검증 |
| **언어** | 한국어 단일 | i18n 구조만 세팅, 번역은 Phase B |
| **오프셋** | 설정 화면에서 수동 수치 입력 | 전용 캘리브레이션 화면 없이 최소 구현 |

#### 차트 에디터 (`apps/editor`)

차트 없이는 게임을 테스트할 수 없으므로, 에디터도 최소 기능이 필요하다.

| 카테고리 | 기능 | 근거 |
|---------|------|------|
| **노트 배치** | Create 모드 — 싱글/롱/트릴/더블 전 타입 | 전 노트 타입이 필요 |
| **노트 편집** | Select 모드(기본 선택 + 이동), Delete 모드(클릭 삭제) | 최소 편집 기능 |
| **마커** | BPM 마커, 박자 마커 배치 + 값 편집 | 차트 타이밍 정의 필수 |
| **파형** | 음원 파형 기본 표시 | BPM 정렬에 필수 |
| **재생** | 재생/정지 | 배치 확인 |
| **스냅** | 고정 스냅 (1/4, 1/8, 1/16) | 정확한 노트 배치 |
| **줌** | 기본 줌 조절 | 밀집 구간 편집 |
| **저장/불러오기** | JSON 파일 다운로드/업로드 | 차트 데이터 보존 |
| **배치 제약** | 4가지 제약 조건 실시간 검증 | 유효하지 않은 차트 방지 |
| **배포** | 로컬 실행만 (`pnpm dev --filter editor`) | 개발자 전용 |

#### 공유 패키지 (`packages/shared`)

| 모듈 | 내용 |
|------|------|
| `types/` | ChartData, NoteType, BpmMarker, TimeSignatureMarker, Beat |
| `chart/` | 분수 연산, beat→time 변환, 배치 제약 검증 |
| `constants/` | 노트 타입 enum, 판정 윈도우 |

### 제외 기능 (Phase B로 이동)

| 기능 | 제외 이유 |
|------|----------|
| Google 로그인 | 메커니즘 검증에 인증 불필요. 게스트로 충분 |
| 랭킹 | 내부 테스트 단계에서 경쟁 시스템 불필요 |
| 튜토리얼 (인게임) | 테스터는 직접 브리핑 가능. "How to Play" 텍스트 안내 1페이지로 대체 |
| 옵저버 모드 | 고난이도 손배치 계획 도구 — Lv.1~6 테스트 차트에서 불필요 |
| 오프셋 캘리브레이션 전용 화면 | 수동 수치 입력으로 대체 |
| 미러 옵션 | 왼손잡이 지원 — 테스트 단계에서 후순위 |
| 곡 프리뷰 재생 | 3~5곡이면 프리뷰 없이 곡 파악 가능 |
| 일시정지 (노트 가림) | 일시정지 기본 구현은 포함, 노트 가림은 Phase B |
| 스페셜 땡스 | 공개 빌드에서 필요 |
| 일본어/영어 | i18n 구조만 세팅 |
| 에디터 — 멀티 선택/드래그 범위 | 기본 선택으로 충분 |
| 에디터 — 우클릭 삭제/드래그 연속 삭제 | 편의 기능 |
| 에디터 — 오토스크롤 | 수동 스크롤로 대체 |
| 에디터 — Supabase 업로드 | JSON 파일 수동 업로드로 대체 |
| 에디터 — 모바일 터치 | 개발자 전용 도구, PC로 충분 |

### Phase A 화면 흐름

```
타이틀 → [첫 기동] → 프리셋 설정 → 곡 선택
타이틀 → [이후 기동] → 곡 선택
곡 선택 → 플레이 → 결과 → 곡 선택
곡 선택 → 설정(키 바인딩 + 스크롤 속도 + 오프셋 수치) → 곡 선택
```

## Phase B: 알파 (비영리 공개 빌드)

### 목표

외부 유저가 처음 접근해도 게임을 이해하고 플레이할 수 있는 상태. 기록 저장과 경쟁 시스템 도입.

### 추가 기능

#### 게임

| 카테고리 | 기능 |
|---------|------|
| **인증** | Google 로그인 + Anonymous Auth → linkIdentity 업그레이드 |
| **랭킹** | 차트별 달성률 순위 |
| **플레이 기록** | 곡별 최고 기록 저장, 결과 히스토리 |
| **튜토리얼** | Phase 1(연주 방법) + Phase 2(복합 패턴/판정). 첫 기동 시 자동 진입 |
| **옵저버 모드** | 자유 스크롤 + 재생/일시정지 + 스크롤 바 마커 기반 구간 반복 + 속도 조절 |
| **오프셋** | 전용 캘리브레이션 화면 |
| **미러** | 좌우 반전 옵션 |
| **일시정지** | 노트 가림 포함 완전 구현 |
| **곡 프리뷰** | 곡 선택 시 자동 재생 |
| **i18n** | 일본어 + 영어 추가 |
| **스페셜 땡스** | 크레딧 + 라이선스 표기 |
| **설정 동기화** | 로그인 유저의 설정/오프셋 서버 저장 |
| **콘텐츠** | 5~10곡 + 튜토리얼 전용 곡 |

#### 차트 에디터

| 카테고리 | 기능 |
|---------|------|
| **선택** | 멀티 선택, Shift/Alt 조합, 드래그 범위 선택 |
| **삭제** | 우클릭 삭제, 드래그 연속 삭제 |
| **재생** | 오토스크롤, 구간 재생 |
| **스냅** | 자유 스냅 값 |
| **저장** | Supabase Storage 직접 업로드/로드 |

## 리스크와 판단 근거

### Phase A에 전 노트 타입을 포함하는 이유

`overview.md`에 "모든 노트 타입은 Lv.1부터 등장"으로 명시. 싱글만 있는 프로토타입은 not4k의 정체성(트릴/더블)을 검증할 수 없다.

### Phase A에 튜토리얼을 제외하는 이유

튜토리얼 구현 비용이 높다(Phase 1~2 콘텐츠 + 전용 곡 제작). 프로토타입 테스터는 개발자가 직접 브리핑 가능하므로, 코어 게임플레이 완성을 우선한다.

### Phase A에 인증을 제외하는 이유

Supabase Auth 연동은 OAuth 콜백, 세션 관리, RLS 정책 등 부수 작업이 많다. 게스트 플레이만으로 메커니즘 검증이 가능하다.

### Phase B에 옵저버 모드를 포함하는 이유

공개 빌드에서 Lv.7+ 차트를 제공하려면 손배치 계획 도구가 필요하다. `observer-mode.md`의 설계 의도와 일치한다.
