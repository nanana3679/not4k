# not4k — 유예 시간 12ms와 키보드 폴링 레이트 관계 분석

> 본 문서는 홀드 이어잡기의 유예 시간(Grace Period) 12ms가 키보드 폴링 레이트에 따라 어떻게 동작하는지를 분석하고, 구현 시 고려사항을 정의한다.

---

## 1. 배경

### 유예 시간의 정의

`note-system.md`에서 정의된 유예 시간:

> 롱노트 진행 중 해당 레인의 모든 키가 떼어진 상태가 **12ms** 이내이면 홀드가 끊기지 않은 것으로 판정한다.

12ms를 선택한 근거: 일반적인 키 전환(한 키를 떼고 다른 키를 누르는 동작)의 물리적 공백은 5~15ms 범위이며, 12ms는 이 범위의 상단에 해당한다.

### 키보드 폴링 레이트

키보드가 호스트(PC)에게 입력 상태를 보고하는 주기.

| 폴링 레이트 | 보고 간격 | 주요 사용처 |
|-------------|-----------|-------------|
| 125Hz | 8ms | USB 기본값, 구형 키보드 |
| 250Hz | 4ms | 일부 게이밍 키보드 |
| 500Hz | 2ms | 게이밍 키보드 표준 |
| 1000Hz | 1ms | 고급 게이밍 키보드 |
| 4000Hz | 0.25ms | 최신 경쟁용 키보드 (Wooting, Razer 등) |
| 8000Hz | 0.125ms | 극한 경쟁용 키보드 (EZ2ON 권장) |

## 2. 문제 분석

### 2.1 폴링 레이트에 따른 실제 감지 가능한 유예 시간

유예 시간 12ms는 "게임 엔진이 인지하는 릴리즈 지속 시간"이다. 그러나 키보드 폴링 레이트에 따라 릴리즈 이벤트의 보고 시점이 양자화(quantized)된다.

#### 125Hz 키보드 (8ms 간격)

```
시간:  0ms    8ms    16ms    24ms
키보드: ─────┼──────┼───────┼──────
        ^보고  ^보고   ^보고    ^보고
```

유저가 실제로 5ms 동안만 키를 떼었더라도:

- **최선의 경우**: 릴리즈가 보고 직후 발생하고 다음 보고 전에 재입력 → 게임 엔진은 0ms 공백으로 인지
- **최악의 경우**: 릴리즈가 보고 직전에 발생하고, 다음 보고 시점에서야 릴리즈가 감지된 뒤, 그 다음 보고까지 재입력이 감지되지 않음 → 게임 엔진은 최대 **16ms** 공백으로 인지

```
실제 동작:     A(release)──────────B(press)
                     5ms 실제 공백

125Hz 감지:   [보고]────8ms────[보고]────8ms────[보고]
              A=held          A=released       B=held

게임 인지:           8~16ms 공백으로 인지
```

12ms 유예 시간에서 125Hz 키보드는 실제 5ms 공백이 최대 16ms로 인지될 수 있어, **유예 시간을 초과하는 거짓 실패(false failure)**가 발생할 수 있다.

#### 1000Hz 키보드 (1ms 간격)

```
실제 동작:     A(release)──────────B(press)
                     5ms 실제 공백

1000Hz 감지:  매 1ms마다 보고

게임 인지:           5~6ms 공백으로 인지 (거의 정확)
```

1000Hz 키보드에서는 실제 공백과 게임 인지 공백의 오차가 최대 1ms이므로, 12ms 유예 시간이 의도대로 동작한다.

### 2.2 폴링 레이트별 유예 시간의 실효 범위

| 폴링 레이트 | 보고 간격 | 12ms 유예 시간의 실효 범위 | 거짓 실패 가능성 |
|-------------|-----------|---------------------------|-----------------|
| 125Hz | 8ms | 4ms~20ms (±8ms 오차) | **높음** — 8ms 실제 공백이 16ms로 인지될 수 있음 |
| 250Hz | 4ms | 8ms~16ms (±4ms 오차) | **중간** — 경계값 근처에서 발생 가능 |
| 500Hz | 2ms | 10ms~14ms (±2ms 오차) | **낮음** — 대부분의 정상 이어잡기가 통과 |
| 1000Hz | 1ms | 11ms~13ms (±1ms 오차) | **매우 낮음** — 거의 정확 |
| 4000Hz+ | 0.25ms | ~12ms (오차 무시 가능) | **없음** |

### 2.3 브라우저 환경의 추가 제약

not4k는 웹 게임이므로 브라우저의 이벤트 처리 파이프라인을 경유한다.

#### KeyboardEvent 타임스탬프

- `KeyboardEvent.timeStamp`는 고해상도 타이머(`performance.now()` 기반)로 마이크로초 정밀도를 제공한다 (대부분의 최신 브라우저).
- 그러나 이 타임스탬프는 **이벤트가 브라우저에 도달한 시점**이지, **물리적 키 입력이 발생한 시점**이 아니다.
- 브라우저의 이벤트 루프 지연, 자바스크립트 실행 지연 등이 추가될 수 있다.

#### requestAnimationFrame 기반 처리

게임 루프가 `requestAnimationFrame`(일반적으로 60Hz=16.67ms 간격)으로 동작하는 경우:

```
실제 키 입력:  ──A(release)───B(press)──
                      5ms 공백

rAF 프레임:    [frame]────16.67ms────[frame]────16.67ms────[frame]

프레임 내 처리: 하나의 프레임에서 A=released, B=pressed를 모두 감지 → 공백 0ms
또는:          프레임 경계를 걸쳐 A=released만 감지 → 다음 프레임에서 B=pressed → 공백 16.67ms
```

rAF 기반에서는 유예 시간 12ms가 사실상 **0ms 또는 16.67ms**로 이분화되어, 12ms라는 세밀한 값의 의미가 퇴색한다.

## 3. 설계 결정

### 3.1 유예 시간의 측정 기준

**유예 시간은 KeyboardEvent의 고해상도 타임스탬프를 기준으로 측정한다.**

- `keyup` 이벤트의 `timeStamp`와 `keydown` 이벤트의 `timeStamp`의 차이로 공백을 계산한다.
- 게임 루프의 프레임 주기에 의존하지 않는다.
- 이벤트를 큐에 쌓고, 각 프레임에서 큐를 처리할 때 타임스탬프 기반으로 유예 시간을 판정한다.

```
// 의사코드
event_queue: Queue<{type: 'keydown'|'keyup', key: KeyId, timestamp: DOMHighResTimeStamp}>
lane_keys_held: Map<Lane, Set<KeyId>>  // 레인별 현재 눌린 키
lane_release_timestamp: Map<Lane, DOMHighResTimeStamp | null>
GRACE_PERIOD_MS = 12
GRACE_PERIOD_TIMEOUT_MS = 1000  // 너무 오래된 릴리즈 정보 무효화

function process_events_for_frame(current_time):
    while event = event_queue.dequeue():
        lane = get_lane(event.key)
        if lane == null: continue

        if event.type == 'keyup':
            lane_keys_held[lane].remove(event.key)
            // 해당 레인의 모든 키가 떨어졌을 때만 릴리즈 타임스탬프 기록
            if lane_keys_held[lane].is_empty():
                lane_release_timestamp[lane] = event.timestamp

        if event.type == 'keydown':
            lane_keys_held[lane].add(event.key)
            if lane_release_timestamp[lane] != null:
                gap = event.timestamp - lane_release_timestamp[lane]
                if gap > GRACE_PERIOD_TIMEOUT_MS:
                    // 너무 오래된 릴리즈 → 무효화
                    lane_release_timestamp[lane] = null
                elif gap <= GRACE_PERIOD_MS:
                    // 홀드 이어잡기 성공
                else:
                    // 홀드 끊김
                lane_release_timestamp[lane] = null
```

이 방식의 이점:

- rAF 프레임 주기(16.67ms)에 의한 양자화를 회피한다
- 키보드 폴링 레이트의 영향은 여전히 존재하지만, 게임 루프의 영향은 제거된다
- `performance.now()` 기반 타임스탬프는 서브밀리세컨드 정밀도를 제공한다

### 3.2 폴링 레이트 불이익에 대한 대응

125Hz 키보드 사용자에게 발생할 수 있는 거짓 실패에 대해 다음 전략을 채택한다.

#### 방안: 유예 시간 12ms를 유지하되, 문서화로 대응

- 12ms는 물리적 공백 5~15ms의 상단을 커버하는 값으로, 설계 의도가 명확하다.
- 125Hz 키보드에서의 거짓 실패는 하드웨어 한계에 의한 것이며, 게임이 이를 보정하면 고폴링 키보드 사용자에게 역불이익(의도적 릴리즈가 이어잡기로 처리됨)이 발생한다.
- 대부분의 게이밍 키보드는 1000Hz를 지원하며, not4k의 타겟 유저(기존 리듬게임 유저)는 게이밍 키보드를 사용할 가능성이 높다.
- 설정 화면에서 "키보드 폴링 레이트가 낮을 경우 홀드 이어잡기가 불안정할 수 있습니다" 안내를 표시한다.

#### 기각된 방안: 폴링 레이트에 따른 동적 유예 시간

```
// 기각: 공정성 문제
grace_period = base_grace + (1000 / polling_rate)
```

- 저폴링 키보드에 보정을 적용하면, 고폴링 키보드 사용자와의 공정성이 깨진다.
- 폴링 레이트를 정확히 감지하는 것이 브라우저 환경에서 불가능하거나 부정확하다.
- 리듬게임에서 하드웨어 성능 차이에 의한 미세한 불이익은 업계 관례상 수용된다 (EZ2ON의 8000Hz 권장, osu!의 고주사율 모니터 권장 등).

## 4. 폴링 레이트와 판정 윈도우의 관계

유예 시간 외에도, 폴링 레이트는 일반 판정 윈도우(Perfect ±41ms 등)에도 영향을 미친다.

| 판정 등급 | 윈도우 (±ms) | 125Hz 오차 (±8ms) | 1000Hz 오차 (±1ms) |
|-----------|-------------|-------------------|-------------------|
| Perfect | ±41ms | 실효 ±33~49ms | 실효 ±40~42ms |
| Great | ±82ms | 실효 ±74~90ms | 실효 ±81~83ms |
| Good | ±120ms | 실효 ±112~128ms | 실효 ±119~121ms |
| Bad | ±160ms | 실효 ±152~168ms | 실효 ±159~161ms |

Perfect 윈도우(±41ms)에서 125Hz의 ±8ms 오차는 윈도우 폭의 약 20%에 해당하여 유의미한 영향을 미친다. 다만 이는 리듬게임 업계에서 일반적으로 수용되는 범위이며 (DJMAX의 100% 판정도 ±41.7ms이고, PC 환경의 다양성을 전제로 설계됨), 유예 시간 12ms에서의 영향이 상대적으로 더 심각하다.

## 5. 결론 및 권장사항

### 확정 사항

1. **유예 시간 12ms를 유지한다.** 설계 의도(물리적 공백 5~15ms 커버)에 부합한다.
2. **KeyboardEvent 타임스탬프 기반으로 측정한다.** rAF 프레임 주기에 의존하지 않는다.
3. **폴링 레이트에 따른 동적 보정은 하지 않는다.** 공정성과 감지 불가능성 문제.

### 구현 시 주의사항

1. **이벤트 큐 처리**: `keydown`/`keyup` 이벤트를 즉시 큐에 넣고, 게임 루프에서 타임스탬프 기반으로 처리한다.
2. **타임스탬프 정밀도 확인**: `event.timeStamp`가 `DOMHighResTimeStamp`(마이크로초 정밀도)인지 확인한다. 일부 구형 브라우저에서는 밀리세컨드 정밀도만 제공할 수 있다.
3. **테스트**: 125Hz, 1000Hz 키보드에서 홀드 이어잡기 성공률을 비교 테스트한다.

### 문서 반영 사항

`note-system.md`의 유예 시간 섹션에 다음을 추가한다:

> **측정 기준**: 유예 시간은 키보드 이벤트의 고해상도 타임스탬프(`KeyboardEvent.timeStamp`)를 기준으로 측정하며, 게임 루프의 프레임 주기에 의존하지 않는다. 키보드 폴링 레이트가 낮을 경우(125Hz 등) 실제 공백보다 길게 감지될 수 있으나, 대부분의 게이밍 키보드(1000Hz)에서는 정확하게 동작한다.
