# 최고판정 노트 리서치 리포트

> 입력 타이밍과 무관하게, 입력을 하기만 하면 최고 판정을 부여하는 노트의 기존 사례와 not4k에서의 구현 방안을 분석한다.

---

## 1. 용어 정의

**최고판정 노트**: 유저가 판정 윈도우 내에서 입력을 수행하기만 하면, 타이밍 정밀도와 무관하게 최고 등급(Perfect)을 부여하는 노트. 일반 노트의 판정 윈도우 세분화(Perfect/Great/Good/Bad)를 적용하지 않고, Bad 윈도우 내의 어떤 입력이든 최고 판정으로 처리한다.

---

## 2. 기존 리듬게임의 사례

### 2.1. 츄니즘 — ExTAP

츄니즘의 ExTAP(황금색 노트)은 판정 범위가 넓어 Justice Critical(최고 판정)을 받기 용이하도록 설계된 노트이다. 일반 TAP보다 관대한 판정 윈도우를 가지며, 유저가 대략적인 타이밍에 입력하기만 하면 최고 판정을 받을 수 있다.

- **설계 의도**: 고난도 곡에서 유저가 좌절하지 않고 하이라이트를 즐길 수 있도록 정확도 부담을 완화한다. 시각적으로 화려하게 연출되어 곡의 강조 지점에서 쾌감을 극대화한다.
- **판정 구조**: 완전히 "입력만 하면 최고 판정"은 아니지만, 판정 윈도우가 매우 넓어 실질적으로 거의 모든 입력이 최고 판정을 받는다.
- **스코어링 영향**: 달성률 1,010,000점 체계에서 일반 노트와 동일한 배점을 가지지만, 최고 판정 획득이 쉬우므로 달성률 확보에 유리하다.

### 2.2. maimai — 슬라이드 노트

maimai의 슬라이드 노트는 시작점(Star)을 탭한 후 궤적을 따라 손을 움직이는 노트이다. 궤적 추적 판정은 매우 관대하여, 주요 거점 센서만 제시간에 통과하면 최고 판정(Critical Perfect)을 부여한다.

- **설계 의도**: 슬라이드의 본질은 공간 이동의 쾌감이지, 타이밍 정밀도가 아니다. 궤적을 완벽하게 따라가지 않아도 판정을 부여함으로써, 관절의 무리한 꺾임을 방지하는 인체공학적 배려와 동시에 퍼포먼스적 유희를 제공한다.
- **판정 구조**: 궤적의 주요 거점을 통과했는지 여부로 판정하며, 통과 타이밍의 정밀도는 거의 요구하지 않는다. 사실상 "입력(궤적 추적)을 하기만 하면 최고 판정"에 해당한다.
- **스코어링 영향**: 슬라이드 자체의 배점은 높지 않으며, 시작점(Star)의 탭 판정이 별도로 존재한다.

### 2.3. 프로젝트 세카이 — 슬라이드 노트

프로젝트 세카이(프로세카)의 슬라이드 노트는 시작점에서 손가락을 올린 후 궤적을 따라가는 노트이다. 슬라이드의 중간 경로에서는 타이밍 판정이 거의 없으며, 경로 위에 손가락이 있기만 하면 최고 판정을 부여한다.

- **설계 의도**: 모바일 환경에서 터치 슬라이드의 자연스러운 UX를 유지하면서, 복잡한 궤적을 따라가는 시각적 재미에 집중할 수 있도록 한다.
- **판정 구조**: 슬라이드 경로의 체크포인트 통과 여부로 판정. 정밀한 타이밍은 요구하지 않는다.

### 2.4. DJMAX — 자동(오토) 노트 / 사이드 트랙

DJMAX RESPECT V의 특정 모드에서 사이드 트랙(Side Track) 노트는 매우 관대한 판정을 가지며, 레인 밖에서 독립적으로 동작한다. 또한 일부 곡의 특수 연출 노트는 입력만으로 최고 판정을 부여하는 구조를 가진다.

- **설계 의도**: 유저의 주의를 메인 레인에 집중시키면서, 사이드 트랙은 부가적인 점수 소스로 기능한다. 사이드 트랙의 판정이 까다로우면 메인 패턴 처리에 지장을 주므로 판정을 관대하게 설정한다.

### 2.5. osu! — 스피너 / 슬라이더

osu! (스탠다드)의 슬라이더는 시작점만 정밀한 타이밍 판정이 있고, 슬라이더 경로를 따라가는 동안은 커서가 경로 위에 있기만 하면 판정을 유지한다. 스피너는 정해진 횟수만큼 회전하기만 하면 판정을 받는다.

- **판정 구조**: 슬라이더의 시작점은 일반 히트서클과 동일한 타이밍 판정이 적용되지만, 슬라이더의 몸통과 끝점은 "존재하기만 하면" 판정을 부여한다.

### 2.6. 비트매니아 IIDX — Charge Note의 끝점 판정

비트매니아 IIDX의 Charge Note(CN)는 시작점에서 키를 누르고, 끝점에서 키를 떼는 구조이다. 끝점의 릴리즈 판정은 상대적으로 관대하여, 일정 범위 내에서 떼기만 하면 최고 판정 또는 그에 준하는 판정을 부여한다.

- **not4k와의 유사성**: not4k의 롱노트 종결 판정도 동일한 구조이다. Good 이상의 릴리즈는 모두 Perfect로 상향되므로, 끝점 판정은 사실상 "떼기만 하면 최고 판정"에 해당한다.

### 2.7. 사례 요약

| 게임 | 노트 유형 | 판정 관대함의 수준 | 핵심 설계 의도 |
|------|-----------|-------------------|---------------|
| 츄니즘 | ExTAP | 판정 윈도우 확대 | 정확도 부담 완화, 시각적 쾌감 |
| maimai | 슬라이드 | 거점 통과만 확인 | 공간 이동의 쾌감, 인체공학적 배려 |
| 프로세카 | 슬라이드 | 경로 위 존재 확인 | 터치 UX, 시각적 재미 |
| DJMAX | 사이드 트랙 | 관대한 윈도우 | 메인 레인 집중 |
| osu! | 슬라이더 몸통 | 커서 존재 확인 | 에이밍과 타이밍의 분리 |
| IIDX | CN 끝점 | 관대한 릴리즈 판정 | 홀드 자체에 집중 |

**공통 패턴**: 최고판정 노트는 대부분 **"입력의 존재 여부"만 확인하고 "입력의 정밀도"는 요구하지 않는** 구조이다. 이를 통해 해당 노트가 가진 본래 목적(궤적 추적, 공간 이동, 홀드 유지 등)에 유저가 집중할 수 있도록 한다.

## 3. not4k 현재 판정 시스템 분석

### 3.1. 판정 구조 요약

not4k의 판정은 노트 타입에 따라 다음과 같이 작동한다.

| 노트 타입 | 판정 이벤트 | 판정 등급 |
|-----------|------------|----------|
| 싱글 | keydown 타이밍 | Perfect/Great/Good/Bad/Miss |
| 더블 | keydown 타이밍 × 2 | 독립된 싱글 판정 × 2 |
| 트릴 | keydown 타이밍 + 교대 확인 | Perfect/Great/Good/Bad/Miss + Good◇ |
| 롱노트 (종결) | keyup 타이밍 | Good 이상 → Perfect 상향 |
| 롱노트 (연결) | 홀드 유지 확인 | Perfect 또는 Miss |

### 3.2. 이미 존재하는 "최고판정" 요소

not4k에는 이미 최고판정에 해당하는 판정 규칙이 존재한다.

1. **롱노트 종결 판정**: Good 이상의 릴리즈는 모두 Perfect로 상향된다. 즉, Bad 윈도우(±160ms) 내에서 Good 윈도우(±120ms) 안에 릴리즈하기만 하면 Perfect이다. 사실상 "떼기만 하면 최고 판정"이다.
2. **롱노트 연결 판정**: 끝점 시점에 해당 레인에 홀드가 유지되고 있으면 Perfect이다. "누르고 있기만 하면 최고 판정"이다.

### 3.3. 누락된 요소: keydown 기반 최고판정 노트

현재 not4k에는 **keydown 입력을 하기만 하면 최고 판정을 주는 노트 타입이 존재하지 않는다**. 싱글/더블/트릴 노트는 모두 타이밍 정밀도에 따른 판정 세분화(Perfect~Bad)를 적용한다.

## 4. 길이 0 롱노트를 이용한 최고판정 노트 구현 시도와 문제점

### 4.1. 아이디어

not4k의 롱노트 종결 판정은 Good 이상이면 Perfect로 상향된다. 따라서 길이 0인 롱노트(시작점 = 끝점)를 배치하면, 바디 유지 구간이 없으므로 유지 판정을 생략하고 keyup 이벤트만으로 종결 판정을 받을 수 있다. 이 판정에서 Good 이상은 Perfect로 상향되므로, "떼기만 하면 최고 판정"인 노트가 된다.

현재 코드(`JudgmentEngine.ts` 319~322행)에서 이 동작이 구현되어 있다:

```typescript
// 길이 0인 롱노트는 바디 홀드 없이 바로 릴리즈 대기
const noteEndTime = this.noteEndTimesMs.get(i);
if (noteEndTime !== undefined && noteEndTime === noteTime) {
  this.noteStates.set(i, NoteState.BODY_AWAITING_RELEASE);
}
```

### 4.2. 문제점: 입력 이벤트 라우팅

**핵심 문제**: 롱노트(RangeNote)는 `findEarliestUnprocessedNote`에서 입력 대상에서 제외된다.

```typescript
// JudgmentEngine.ts 351행
// 바디 노트(RangeNote)는 입력 대상이 아님 — 자동 활성화로 처리
if ("endBeat" in note) continue;
```

이 설계에 의해 길이 0 롱노트 위에서 keydown을 눌러도, 해당 이벤트는 롱노트를 건너뛰고 같은 레인의 다음 포인트 노트(싱글/더블/트릴)에 전달된다. 결과적으로:

1. 유저가 길이 0 롱노트 위치에서 키를 눌렀을 때, keydown 이벤트가 다른 노트에 소비된다.
2. 길이 0 롱노트는 `BODY_AWAITING_RELEASE` 상태에서 keyup을 기다리지만, keydown이 다른 노트에 의해 "소비"되면서 유저의 입력 의도가 왜곡된다.
3. 특히 길이 0 롱노트와 같은 위치에 싱글 노트가 있으면, keydown은 싱글 노트에, keyup은 길이 0 롱노트에 각각 전달되어 의도대로 동작하지만, **길이 0 롱노트 단독으로는 keydown 이벤트를 "차단"하지 못하여** 뒤에 있는 노트가 의도치 않게 판정된다.

### 4.3. 설계 수준의 제약

이 문제는 단순한 버그가 아니라 **아키텍처 수준의 제약**이다.

- 롱노트는 의도적으로 입력 매칭 대상에서 제외되어 있다. 롱노트는 "자동 활성화"(시작 시간 도달 시 상태 전환)로 처리되며, keydown 이벤트를 소비하지 않는다.
- 이 설계는 "헤드 노트와 롱노트는 별개의 엔티티"라는 핵심 원칙(`note-system.md`)에 기반한다. 헤드 노트가 keydown을 소비하고, 롱노트는 held 상태만 체크한다.
- 길이 0 롱노트를 keydown 소비 대상으로 만들면, 기존 롱노트의 입력 라우팅 규칙과 충돌한다.

## 5. 구현 방안 제안

### 5.1. 방안 A: 새로운 포인트 노트 타입 추가

가장 직접적인 해결책으로, `NoteType`에 새로운 타입(예: `AUTO_PERFECT` 또는 `IMPACT`)을 추가한다.

**구현 개요**:
- `note.ts`에 `NoteType.AUTO_PERFECT = "autoPerfect"` 추가
- `POINT_NOTE_TYPES`에 포함 (포인트 엔티티이므로 keydown 매칭 대상)
- `JudgmentEngine.processSingleNoteInput`과 유사한 핸들러를 추가하되, `calculateGrade` 대신 항상 `JudgmentGrade.PERFECT`를 반환
- 판정 윈도우는 Bad(±160ms)를 그대로 사용하여, 윈도우 내 입력이면 무조건 Perfect

**장점**:
- 기존 판정 시스템의 아키텍처를 변경하지 않음
- `findEarliestUnprocessedNote`가 자연스럽게 이 노트를 매칭 대상으로 포함
- 차트 에디터에서 독립적인 노트 타입으로 배치 가능
- 시각적으로 다른 노트와 구분 가능 (색상/형태 차별화)

**단점**:
- 새로운 노트 타입 추가에 따른 코드 변경 범위가 넓음 (렌더러, 에디터, 직렬화 등)
- 노트 타입이 늘어나면 시스템 복잡도가 증가

**스코어링 고려사항**:
- 이 노트는 항상 Perfect(3점)이므로, 달성률에 기여하지만 실력 차별화에는 기여하지 않는다
- maimai의 Break 노트처럼 곡의 강조점에 배치하여 "임팩트 순간"을 만드는 용도로 활용 가능

### 5.2. 방안 B: 싱글 노트에 "최고판정 플래그" 추가

기존 싱글 노트 엔티티에 `autoPerfect: boolean` 플래그를 추가한다.

**구현 개요**:
- `NoteEntity` 타입에 선택적 필드 `autoPerfect?: boolean` 추가
- `processSingleNoteInput`에서 `note.autoPerfect === true`이면 `calculateGrade` 대신 `JudgmentGrade.PERFECT` 사용
- 차트 JSON에서 기존 싱글 노트에 플래그만 추가

**장점**:
- 새로운 노트 타입을 추가하지 않으므로 변경 범위가 작음
- 기존 싱글 노트의 모든 인프라(렌더링, 매칭, 직렬화)를 그대로 재사용
- 더블/트릴 노트에도 동일한 플래그를 적용할 수 있어 확장성이 있음

**단점**:
- 시각적 구분이 노트 타입이 아닌 플래그에 의존하므로, 렌더러에서 추가 분기 필요
- 노트 타입의 의미가 흐려질 수 있음 (싱글인데 판정이 다른 노트)

### 5.3. 방안 C: 길이 0 롱노트의 입력 라우팅 수정

기존 길이 0 롱노트를 살리되, 입력 라우팅 문제를 해결한다.

**구현 개요**:
- 길이 0 롱노트가 `BODY_AWAITING_RELEASE` 상태일 때, `onLanePress`에서 해당 노트 위치의 keydown 이벤트를 "차단"하는 로직 추가
- 또는 길이 0 롱노트를 keydown 기반으로 전환: `findEarliestUnprocessedNote`에서 길이 0 롱노트는 포인트 노트처럼 매칭하되, 판정은 항상 Perfect

**장점**:
- 새로운 노트 타입이나 플래그를 추가하지 않음
- "헤드 없는 롱노트의 길이가 0이면 최고판정 노트"라는 직관적인 규칙

**단점**:
- `findEarliestUnprocessedNote`의 RangeNote 제외 규칙에 예외를 만들어야 함
- 롱노트의 "자동 활성화 → keyup 기반 판정"이라는 일관된 모델이 깨짐
- 길이 0 롱노트가 keydown을 소비하면, 같은 위치의 다른 포인트 노트와의 상호작용이 복잡해짐
- 기존 아키텍처 원칙("롱노트는 입력 이벤트를 소비하지 않는다")과 충돌

### 5.4. 방안 비교

| 기준 | A: 새 노트 타입 | B: 플래그 추가 | C: 길이 0 수정 |
|------|-----------------|---------------|---------------|
| 아키텍처 일관성 | 높음 | 보통 | 낮음 |
| 구현 범위 | 넓음 | 좁음 | 중간 |
| 확장성 | 높음 | 높음 | 낮음 |
| 기존 시스템 영향 | 없음 | 낮음 | 있음 |
| 시각적 구분 | 자연스러움 | 추가 작업 필요 | 추가 작업 필요 |
| 차트 에디터 통합 | 독립적 | 기존 노트에 옵션 | 기존 노트 조합 |

### 5.5. 권장 방안

**방안 B (싱글 노트에 최고판정 플래그 추가)를 권장한다.**

근거:

1. **최소 변경 원칙**: 새로운 노트 타입을 추가하면 렌더러, 에디터, 직렬화, 스킨 시스템 등 광범위한 수정이 필요하다. 플래그 추가는 판정 로직의 한 분기만 수정하면 된다.
2. **기존 아키텍처 유지**: 롱노트의 입력 라우팅을 건드리지 않으므로, 핵심 설계 원칙("롱노트는 입력 이벤트를 소비하지 않는다")이 보존된다.
3. **확장 가능**: 더블 노트나 트릴 노트에도 동일한 플래그를 적용할 수 있다. 예를 들어, 더블 노트에 `autoPerfect` 플래그를 설정하면 "양손으로 누르기만 하면 최고 판정"인 임팩트 노트가 된다.
4. **시각적 구분**: 렌더러에서 `autoPerfect` 플래그가 있는 노트에 별도의 색상이나 이펙트를 적용하면 충분하다. maimai의 Break 노트(빨강/노랑)나 츄니즘의 ExTAP(황금색)처럼, 기존 노트와 같은 형태이되 색상으로 구분하는 접근이 자연스럽다.
5. **스코어링 의미**: 이 노트는 "이 지점에서는 타이밍이 아닌 입력 자체가 중요하다"는 차트 디자이너의 의도를 명확히 전달한다. scoring.md의 설계 방향("더블 노트는 가중치를 가진다")과 마찬가지로, 최고판정 노트는 "반드시 입력해야 하는 순간"을 만드는 도구가 된다.

## 6. 활용 시나리오

### 6.1. 곡의 하이라이트 강조

maimai의 Break 노트처럼, 곡의 하이라이트에 최고판정 노트를 배치하여 유저가 "이 순간은 놓치지 않고 입력했다"는 성취감을 느끼게 한다. 타이밍 부담을 제거하여 음악적 순간에 집중할 수 있다.

### 6.2. 학습 보조

입문 등급(Lv.1~3)에서 새로운 입력방식(트릴, 더블 등)이 처음 등장할 때, 최고판정 노트를 사용하여 "일단 입력하기만 하면 성공"이라는 경험을 제공한다. 이후 일반 판정 노트로 전환하여 타이밍 정밀도를 요구하는 단계적 학습을 유도한다.

### 6.3. 릴리즈탭 진입

헤드 없는 롱노트의 종결 판정은 이미 최고판정(Good 이상 → Perfect 상향)이므로, keydown 쪽에도 최고판정 노트를 배치하면 "누르기 + 떼기" 한 쌍이 모두 최고판정이 되어, 릴리즈탭 학습의 진입 장벽을 낮출 수 있다.

## 7. 결론

1. 기존 리듬게임에서 "입력만 하면 최고 판정"인 노트는 광범위하게 사용되고 있으며, 주로 **입력 부담 완화**, **음악적 강조**, **학습 보조**의 목적으로 설계된다.
2. not4k에서 길이 0 롱노트를 이용한 최고판정 노트 구현은, 롱노트가 keydown 이벤트를 소비하지 않는 아키텍처 제약으로 인해 실패한다. 이 문제를 해결하려면 롱노트의 핵심 설계 원칙을 변경해야 하므로 비용이 크다.
3. **기존 포인트 노트(싱글/더블/트릴)에 `autoPerfect` 플래그를 추가하는 방안**이 가장 적은 변경으로 최고판정 노트를 구현할 수 있다. 기존 입력 매칭 인프라를 그대로 활용하면서, 판정 계산 단계에서만 분기하면 된다.
