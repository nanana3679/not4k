## 1. 현재 Zoom 구현 분석

### 1.1 줌 파라미터 구조

현재 줌은 **연속적인(continuous) `pixelPerSecond` 값**으로 구현되어 있다.

| 항목 | 값 |
|------|-----|
| 타입 | `number` (실수) |
| 의미 | 1초에 해당하는 픽셀 수 (px/s) |
| 기본값 | 200 |
| 최솟값 | 50 (`SnapZoomController.MIN_ZOOM`) |
| 최댓값 | 2000 (`SnapZoomController.MAX_ZOOM`) |
| 변경 방식 | `Ctrl + 휠` — 1.1배씩 곱/나눗셈 (`ZOOM_FACTOR = 1.1`) |

줌 값은 세 곳에서 관리된다:
- **EditorStore** (`zoom: number`): React 상태의 단일 소스
- **SnapZoomController** (`state.zoom`): 입력 처리 및 클램핑
- **TimelineRenderer** (`_zoom`): 렌더링에 사용되는 로컬 복사본

### 1.2 줌이 렌더링에 미치는 영향

줌 값이 변경되면 `TimelineRenderer.render()`가 호출되어 **전체 타임라인을 처음부터 다시 그린다**:

```
zoom setter → render() → clearDynamicLayers() → 모든 레이어 재구축 → app.render()
```

`render()` 내부에서 실행되는 작업:

1. **`clearDynamicLayers()`** — 18개 Container의 `removeChildren()` 호출
2. **`renderLaneBackgrounds()`** — 레인 배경 Graphics 생성 (4~N개)
3. **`renderWaveform()`** — 웨이브폼 폴리곤 전체 재구축 (피크 수만큼 lineTo)
4. **`renderGridLines()`** — 마디선 + 비트선 + 스냅선 + Text 라벨 생성 (마디 수 * 스냅 수)
5. **`renderTrillZones()`** — 트릴 구간 사각형
6. **`renderMoveOrigins()`** — 이동 중 원본 위치 고스트
7. **`renderBoxSelectRect()`** — 범위 선택 사각형
8. **`renderNotes()`** — 모든 노트를 순회하며 Graphics 생성 (노트 수 비례)
9. **`renderMarkers()`** — 이벤트 마커 + Text 라벨
10. **`updateScroll()`** — 모든 레이어 위치 갱신
11. **`renderMinimap()`** — 미니맵 전체 재구축
12. **`app.render()`** — PixiJS 최종 draw call

### 1.3 주요 성능 병목

| 병목 지점 | 설명 |
|-----------|------|
| **Graphics 객체 전량 재생성** | 매 `render()` 호출 시 기존 Graphics를 모두 파괴하고 새로 생성. `new Graphics()` 호출이 약 137회 이상 (노트/마커/그리드 합산) |
| **웨이브폼 재구축** | `Float32Array` 피크 데이터를 매번 순회하며 `lineTo()` 폴리곤을 재구축. 3분 곡 기준 약 15,000개 이상의 정점 |
| **그리드선 재계산** | 마디/비트/스냅선을 모든 마디에 대해 반복. 스냅 48분음표 기준 수백 개의 Graphics 생성 |
| **Text 객체 재생성** | 마디 라벨, 이벤트 텍스트 등의 `new Text()` 호출. 텍스트 렌더링은 GPU에서 비용이 높음 |
| **GC 압력** | 매 렌더 사이클마다 대량의 단기 객체 생성/파괴로 가비지 컬렉션 부담 발생 |

### 1.4 줌 변경 빈도

`Ctrl + 휠`은 OS/브라우저에 따라 초당 10~60회의 이벤트를 발생시킨다. 각 이벤트마다 1.1배씩 줌이 변경되므로, 빠르게 휠을 돌리면 **초당 10~60번의 전체 렌더가 발생**할 수 있다. 현재는 디바운스/스로틀이 없다.

## 2. 프리렌더링 접근법 평가

### 2.1 제안된 방식 요약

> "지원하는 배율을 몇 개 정해놓고 미리 렌더링하여 사용감을 개선"

이 방식은 **이산적(discrete) 줌 레벨을 사전 정의**하고, 해당 레벨에서의 렌더링 결과를 캐시하여 줌 변경 시 즉시 표시하는 것이다.

### 2.2 장점

| 장점 | 설명 |
|------|------|
| **줌 전환 시 즉각 반응** | 사전 렌더링된 결과를 바로 표시하므로 줌 변경 시 지연 없음 |
| **렌더링 비용 분산** | 사용자가 줌을 변경하기 전에 미리 렌더링해두므로 순간 부하 감소 |
| **일관된 프레임레이트** | 줌 애니메이션 중에도 렌더링 작업이 없으므로 부드러운 전환 가능 |

### 2.3 단점 및 리스크

| 단점 | 심각도 | 설명 |
|------|--------|------|
| **메모리 사용량 급증** | **높음** | 프리렌더링된 텍스처를 여러 줌 레벨만큼 GPU 메모리에 보관해야 함. 3분 곡 기준 zoom=200에서 타임라인 높이가 약 36,000px. 이를 RenderTexture로 저장하면 한 레벨당 수십~수백 MB의 VRAM 필요 |
| **타임라인 전체 렌더 불가** | **높음** | PixiJS RenderTexture/Canvas 텍스처의 최대 크기는 GPU에 따라 4096~16384px로 제한됨. 줌 200에서 3분 곡의 타임라인 높이는 약 36,000px로 단일 텍스처에 담을 수 없음 |
| **차트 수정 시 전부 무효화** | **높음** | 노트 추가/삭제/이동 시 모든 프리렌더링 캐시를 폐기하고 다시 생성해야 함. 에디터 특성상 수정이 매우 빈번하므로 캐시 히트율이 매우 낮음 |
| **구현 복잡도** | 중간 | 타일링, 캐시 관리, 무효화 로직 등 상당한 추가 코드 필요 |
| **줌 값이 이산적으로 제한됨** | 중간 | 연속적 줌의 자유도를 잃게 됨. 사용자가 원하는 정확한 밀도를 설정할 수 없음 |
| **초기 로딩 시간 증가** | 낮음 | 곡 로드 시 여러 줌 레벨을 미리 렌더링하는 시간 소요 |

### 2.4 핵심 문제: 에디터 vs 뷰어

프리렌더링은 **읽기 전용 뷰어**(예: 게임 플레이, 채보 미리보기)에서는 효과적이지만, **에디터**에서는 근본적인 한계가 있다:

- 에디터에서는 **매 조작마다 렌더 상태가 변경**된다 (노트 추가/삭제/이동, 선택 상태 변경, 고스트 노트 표시 등)
- 프리렌더링된 배경 위에 동적 요소(선택 표시, 고스트, 커서 등)를 오버레이하면 **레이어 합성 비용**이 추가됨
- 캐시 무효화가 너무 잦아 프리렌더링의 이점이 사실상 상쇄됨

## 3. 권장 줌 레벨 (프리렌더링 채택 시)

프리렌더링을 진행한다면, 다음과 같은 이산 줌 레벨을 권장한다.

### 3.1 배율 설계 원칙

- 각 단계 간 **약 2배 비율**: 사용자가 확대/축소 시 의미 있는 밀도 변화를 체감
- 최소/최대 범위 유지: 현재 50~2000 범위 커버
- 음악 에디터의 일반적 관례 반영 (DAW들의 줌 단계 참고)

### 3.2 권장 레벨

| 레벨 | px/s | 비율 (기본 대비) | 용도 |
|------|------|------------------|------|
| 1 | 50 | 0.25x | 전체 곡 조감 |
| 2 | 100 | 0.5x | 넓은 범위 확인 |
| 3 | 200 | 1.0x (기본) | 일반 편집 |
| 4 | 400 | 2.0x | 세밀한 편집 |
| 5 | 800 | 4.0x | 고밀도 구간 편집 |
| 6 | 1600 | 8.0x | 극세밀 편집 |

총 6단계. 각 단계가 정확히 2배이므로 텍스처 스케일링 시 품질 저하가 최소화된다.

## 4. 대안적 성능 개선 방안 (권장)

프리렌더링보다 **더 효과적이면서 구현 복잡도가 낮은** 대안들을 우선 적용할 것을 권장한다.

### 4.1 뷰포트 컬링 (Viewport Culling) -- 최우선 권장

**현재 문제**: 화면에 보이지 않는 영역의 노트/그리드도 모두 렌더링한다.

**해결**: 현재 scrollY + canvas height로 보이는 시간 범위를 계산하고, 해당 범위 내의 요소만 렌더링한다.

```typescript
// 의사코드
const visibleTopMs = renderer.yToTime(0);         // 화면 상단의 시간
const visibleBottomMs = renderer.yToTime(canvasHeight); // 화면 하단의 시간

// 이 범위에 해당하는 노트/그리드만 렌더링
notes.filter(n => {
  const ms = beatToMs(n.beat, ...);
  return ms >= visibleBottomMs && ms <= visibleTopMs;
});
```

**효과**: 전체 곡 대비 화면에 보이는 비율만큼 렌더링 비용 절감. 3분 곡에서 화면에 2~3초만 보인다면 약 **95% 이상의 렌더 비용 절감**.

**난이도**: 낮음 (각 render 메서드에 범위 필터 추가)

### 4.2 Graphics 객체 풀링 (Object Pooling)

**현재 문제**: 매 `render()` 호출 시 모든 Graphics를 파괴하고 새로 생성하여 GC 압력이 높다.

**해결**: Graphics 객체 풀을 유지하고, `render()` 시 풀에서 꺼내 재사용한 뒤 반환한다.

```typescript
class GraphicsPool {
  private pool: Graphics[] = [];

  acquire(): Graphics {
    return this.pool.pop() ?? new Graphics();
  }

  release(g: Graphics): void {
    g.clear();
    this.pool.push(g);
  }
}
```

**효과**: GC 압력 대폭 감소, 객체 생성 비용 절감.

**난이도**: 중간 (기존 `removeChildren()` + `new Graphics()` 패턴을 풀 기반으로 변경)

### 4.3 줌 변경 디바운스/스로틀

**현재 문제**: `Ctrl + 휠` 이벤트마다 즉시 전체 렌더가 실행된다.

**해결**: 줌 변경 시 렌더를 `requestAnimationFrame` 단위로 병합(coalesce)한다.

```typescript
private pendingRender = false;

set zoom(value: number) {
  this._zoom = value;
  if (!this.pendingRender) {
    this.pendingRender = true;
    requestAnimationFrame(() => {
      this.pendingRender = false;
      this.render();
    });
  }
}
```

**효과**: 60fps를 초과하는 이벤트 빈도에서 불필요한 렌더 제거.

**난이도**: 매우 낮음

### 4.4 레이어별 변경 감지 (Dirty Flag)

**현재 문제**: 줌 변경 시 그리드, 노트, 웨이브폼, 미니맵 등 모든 레이어를 다시 그린다. 하지만 실제로 줌이 바뀌면 **위치만 변경되는 요소**(노트, 마커)와 **형태 자체가 변경되는 요소**(그리드 간격, 웨이브폼 폴리곤)가 구분 가능하다.

**해결**: 각 레이어에 dirty flag를 두고, 변경된 레이어만 재구축한다.

```typescript
private dirtyFlags = {
  grid: true,
  notes: true,
  waveform: true,
  minimap: true,
  // ...
};
```

**효과**: 노트를 수정하지 않은 줌 변경 시 노트 레이어 재구축 스킵 가능. 단, 줌은 모든 Y 좌표에 영향을 주므로 줌 변경 시에는 대부분 dirty가 된다. **다른 상황(선택 변경, 스크롤 등)에서 더 유효**함.

**난이도**: 중간

### 4.5 정적 레이어 분리 + Container Transform

**현재 문제**: 줌 변경 시 모든 요소의 Y 좌표를 새 `pixelPerSecond`로 재계산한다.

**해결**: 노트와 그리드를 **시간 기반 좌표계** Container에 배치하고, Container의 `scale.y`를 조절하여 줌을 구현한다.

```typescript
// 노트를 시간 기반 Y 좌표로 배치
noteGraphics.y = -timeMs; // 시간(ms)을 직접 Y로

// 줌은 컨테이너 스케일로 적용
timeContainer.scale.y = zoom / 1000;
```

**장점**: 줌 변경 시 **개별 요소 재배치 불필요** -- Container scale만 변경하면 GPU가 처리. 스크롤도 Container position으로 처리.

**단점**: BPM 변속 구간에서 비균일 시간-공간 매핑이 있으므로 단순 선형 스케일로는 처리 불가. 그리드선 두께, 노트 높이 등 줌에 따라 변하지 않아야 할 요소의 별도 처리 필요.

**효과**: 줌 변경이 O(1) 연산으로 축소됨 (가장 큰 성능 개선).

**난이도**: 높음 (시간-공간 매핑 로직 전면 리팩터링 필요)

### 4.6 웨이브폼 LOD (Level of Detail)

**현재 문제**: 웨이브폼을 항상 전체 해상도로 렌더링한다.

**해결**: 줌 레벨에 따라 다른 해상도의 피크 데이터를 사용한다.

```typescript
// 줌이 낮으면 (축소) 피크를 더 거칠게 샘플링
const effectivePeaks = zoom < 200
  ? decimatePeaks(waveformPeaks, Math.ceil(200 / zoom))
  : waveformPeaks;
```

**효과**: 축소 시 정점 수 대폭 감소.

**난이도**: 낮음

## 5. 권장 구현 순서

프리렌더링보다 아래 순서로 점진적 최적화를 적용하는 것을 권장한다. 각 단계에서 프로파일링하여 충분한 성능이 확보되면 이후 단계는 생략할 수 있다.

| 순서 | 방안 | 예상 효과 | 난이도 | 비고 |
|------|------|-----------|--------|------|
| 1 | **rAF 기반 렌더 병합** (4.3) | 불필요한 렌더 제거 | 매우 낮음 | 즉시 적용 가능, 부작용 없음 |
| 2 | **뷰포트 컬링** (4.1) | 렌더 비용 90%+ 절감 | 낮음 | 가장 높은 ROI |
| 3 | **웨이브폼 LOD** (4.6) | 웨이브폼 렌더 비용 절감 | 낮음 | 축소 시 효과 큼 |
| 4 | **객체 풀링** (4.2) | GC 압력 감소 | 중간 | 위 3개 적용 후에도 프레임 드랍 시 |
| 5 | **Container Transform** (4.5) | 줌 변경이 O(1) | 높음 | 근본적 해결, 대규모 리팩터링 |

### 프리렌더링은 언제 적용하는가?

위 최적화를 모두 적용한 후에도 성능이 부족하거나, **게임 플레이 모드 등 읽기 전용 뷰**에서 줌 전환이 필요한 경우 프리렌더링을 고려한다. 이 경우에도:

- **타일 기반 렌더링**: 타임라인을 세로 512~1024px 타일로 분할하여 각 줌 레벨의 타일을 캐시
- **LRU 캐시**: 메모리 제한 내에서 가장 최근 사용된 타일만 유지
- **비동기 생성**: 줌 전환 시 현재 레벨을 즉시 표시하고, 인접 레벨을 Web Worker에서 백그라운드 생성

| 프리렌더링의 구현 비용 | 높음 (타일링, 캐시 관리, 텍스처 크기 제한 대응 등) |
| 프리렌더링의 메모리 비용 | 매우 높음 (줌 레벨 x 타일 수 x VRAM) |
| **권장 방향** | **뷰포트 컬링 + rAF 병합을 우선 적용**. 이 두 가지만으로 현재 병목의 대부분이 해소될 것으로 예상됨 |

현재 렌더러의 가장 큰 문제는 **화면에 보이지 않는 요소까지 모두 렌더링하는 것**이다. 뷰포트 컬링을 적용하면 줌 50에서 3분 곡 기준 렌더 대상이 전체의 약 2~5%로 줄어든다. 이것만으로도 프리렌더링 없이 충분히 부드러운 줌 경험을 제공할 수 있다.

프리렌더링은 **이산 줌 레벨 제약 + 높은 메모리 비용 + 잦은 캐시 무효화**라는 세 가지 근본 문제를 안고 있어, 에디터의 현재 아키텍처에서는 ROI가 낮다. 대안으로 제시한 점진적 최적화를 순서대로 적용하는 것을 권장한다.
